# /create-e2e-test

## 核心目标
基于项目文档创建端到端测试，验证功能完整性并驱动质量改进。通过测试→分析→修复→验证的闭环，确保文档描述与实际功能的一致性。

## 方法指导

### 文档驱动测试策略
- **以现有文档为规范**：README、API文档、功能说明作为测试用例来源
- **真实场景模拟**：从用户角度设计完整的功能流程测试
- **失败深度分析**：区分功能缺失、API问题、界面不匹配、错误处理缺失等不同失败原因

### 智能修复反馈
- **精准问题定位**：识别是代码问题还是文档问题
- **具体修复建议**：提供可执行的代码修复方案或文档调整建议
- **持续验证循环**：修复后重新测试，直到功能真正可用

### 技术栈自适应
- **框架无关性**：根据项目技术栈选择合适的测试工具和方法
- **路径智能发现**：自动识别项目结构，确定测试文件位置和命名规范
- **环境配置检测**：验证测试环境配置，必要时提供设置指导

## 关键原则

### 务实质量标准
- 功能可用性 > 测试覆盖率
- 核心流程完整 > 边缘案例齐全
- 用户价值验证 > 技术指标优化
- 文档真实反映 > 期望功能描述

### 避免测试陷阱
- 禁用Mock数据掩盖功能缺失
- 测试失败必须深度分析根因
- 不以测试通过率衡量成功，以功能可用性为准
- 保持测试与文档的双向同步更新

## 执行参数
`$ARGUMENTS` - 目标功能模块或页面名称，用于确定测试范围和重点
