# 文章吸收器

## 核心目标

评估用户提供的文章（AI/工程/工作流相关），提炼可操作的经验，整合到 Claude Code 工作流体系（CLAUDE.md、commands、skills）中。

## 输入格式

`/learn-article <文章内容或 URL>`

## 执行策略

### Phase 1: 批判性评估

以**资深工程师**视角审视文章，输出紧凑评估：

1. **一句话核心观点**
2. **逐条判断**（每条一行，用 ✅⚠️❌ 标记）：
   - ✅ 有实践验证、逻辑自洽，与现有工作流吻合或有增量
   - ⚠️ 有道理但需要适配我们的上下文
   - ❌ 过度营销、缺乏实证、与我们的经验矛盾
3. **一句话结论**：值得吸收的增量是什么（或"增量不大，建议 `/note` 归档"）

不要在结论中重复上面已说过的分析。直接问用户是否进入 Phase 2。

### Phase 2: 经验提炼与落地方案

将文章中**值得吸收的部分**映射到具体的整合点：

| 整合目标 | 适用内容 | 操作 |
|---------|---------|------|
| `~/.claude/CLAUDE.md` | 通用工作原则、铁律、策略调整 | 修改对应章节 |
| 项目 `CLAUDE.md` | 项目特定的实践建议 | 标注但不直接改 |
| `commands/*.md` | 新的工作流命令 | 创建或修改命令 |
| `skills/*/SKILL.md` | 自动触发的领域知识 | 创建或修改 Skill |
| Memory | 经验教训、决策依据 | 写入记忆文件 |
| 知识库 `/note` | 纯知识性内容，不改变工作流 | 建议用 `/note` 归档 |

对每个建议的变更，输出：
```
📍 目标：[文件路径]
📝 变更：[新增/修改/删除]
💡 内容摘要：[一句话]
🔗 来源观点：[文章中的原始论述]
```

### Phase 3: 用户确认后执行

- 用户逐条确认或调整后，执行变更
- 变更涉及 CLAUDE.md 时，确保不破坏现有结构，融入对应章节
- 变更涉及 commands/skills 时，遵循各自的编写规范

## 约束条件

- **不盲目跟风**：文章观点必须经过批判性评估，不是所有"最佳实践"都适合我们
- **增量整合**：融入现有体系，不推翻已有决策（除非明确优于现有方案）
- **标注来源**：整合的内容注明灵感来源（文章标题或链接），便于追溯
- **最小变更**：能改一行不改一段，能复用不新建
- **Phase 1 必须等确认**：评估结果需用户确认后才进入落地阶段，避免过度改动
- **URL 输入时**：使用 WebFetch 获取文章内容
