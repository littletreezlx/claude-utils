# Gemini UI 方案咨询

## 目标
收集项目上下文和需求信息，生成结构化提示词供用户与 Gemini 交流，获取高质量的产品/UI 设计方案。

## 执行方式

### 第一阶段：需求理解（快速）

1. **理解用户需求** — 1-2 个问题聚焦核心
2. **查阅项目上下文**：
   - 相关数据模型/接口
   - 现有 UI 组件（UI_SHOWCASE.md 如有）
   - 技术栈限制（Flutter/React/等）
3. **绘制 ASCII 草图**（可选）— 帮助 Gemini 理解现有界面

### 第二阶段：生成 Gemini 提示词

输出格式（用户可直接复制）：

```
------- GEMINI PROMPT START -------

# [功能名称] UI 设计咨询

## 背景
[简述项目类型、技术栈、目标用户]

## 当前状态
[现有界面描述 / ASCII 草图]

## 需求
[用户想要实现什么]

## 约束条件
- 技术栈：[Flutter/React/等]
- 现有组件：[可复用的组件列表]
- 设计规范：[如有]

## 数据模型
```
[相关的数据结构，帮助理解业务]
```

## 期望输出
请提供：
1. **交互流程**：用户操作步骤
2. **界面布局**：关键页面的结构描述
3. **状态设计**：界面需要维护哪些状态
4. **边界情况**：空状态、错误状态、加载状态的处理

可以用 ASCII 图、流程图、或结构化描述。

------- GEMINI PROMPT END -------
```

### 第三阶段：衔接说明

提示词输出后，告知用户：

```
请将上述提示词发送给 Gemini，获取 UI 方案后：
1. 直接粘贴 Gemini 的完整回复给我
2. 或告诉我 Gemini 方案的要点

我会基于项目代码实现该方案。
```

## 约束条件

- **快速响应**：理解阶段不超过 2 轮对话
- **中等深度**：收集必要上下文，不过度堆砌
- **格式清晰**：提示词可直接复制使用
- **技术适配**：根据项目技术栈调整约束描述

## 上下文收集清单

| 类型 | 来源 | 优先级 |
|------|------|--------|
| 数据模型 | 相关 Entity/Model 文件 | 高 |
| 现有组件 | UI_SHOWCASE.md | 中 |
| 技术栈 | package.json / pubspec.yaml | 高 |
| 设计规范 | 项目文档 | 低 |
| 现有界面 | 相关页面代码 | 中 |

## 后续衔接

用户从 Gemini 获取方案后：
- 简单功能 → 直接实现
- 复杂功能 → `/feat-discuss` 细化后实现
- 大型功能 → `/todo-huge-task` 任务拆分
