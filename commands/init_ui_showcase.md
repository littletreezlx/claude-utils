# Flutter UI 文档系统初始化

> 深度分析项目后生成 UI 文档体系，而非套用通用模板

## 核心理念
- **理解驱动** - 先深度分析项目，再基于实际生成文档
- **现状优先** - 保留有效文档，补充缺失内容
- **渐进改进** - 分阶段实施，每步基于前一步结果

---

## 🚀 使用方式

```bash
/init_ui_showcase                    # 标准模式（推荐）
/init_ui_showcase --minimal          # 小项目精简模式
/init_ui_showcase --discovery        # 仅页面发现（快速）
/init_ui_showcase --update-existing  # 仅更新现有配置
```

---

## 📋 执行策略

### 第一阶段：深度分析（必须完成）

**项目架构分析**：
- 扫描 lib/ 目录所有 .dart 文件
- 识别页面、组件、功能模块
- 分析状态管理和数据流

**动态页面发现**（可选）：
- 运行 UI 测试获取所有可达页面
- 验证路由完整性
- 生成页面截图

**测试体系评估**：
- 现有集成测试覆盖
- Golden Test 支持情况
- flutter_common 集成状态

### 第二阶段：方案决策

**项目分类**：
| 规模 | 文件数 | 推荐模式 |
|------|--------|---------|
| 小型 | <50 | 保留增强 + 快速实施 |
| 中型 | 50-200 | 保留增强 + 标准实施 |
| 大型 | >200 | 重构整合 + 阶段实施 |

**改进模式**：
- **保留增强** - 现有基础良好，重点补充
- **重构整合** - 现有文档混乱，需重新组织
- **新建补全** - 文档缺失严重，从零开始

### 第三阶段：分阶段实施

**核心文档**（高优先级）：
- `UI_SHOWCASE.md` - 基于实际界面生成
- 专题文档 - 根据现有组件分类
- AI 协作配置 - 集成到 CLAUDE.md

**截图工具**（中优先级）：
- flutter_common 集成
- 项目配置生成
- 测试脚本创建

**测试体系**（中优先级）：
- 动态页面发现测试
- Golden Test 基础
- CI/CD 集成

### 第四阶段：验证优化

- Claude UI 定位效率测试
- 文档准确性检查
- 建立长期维护计划

---

## 💎 严格禁止

1. **跳过深度分析** ⛔ - 必须先分析再生成
2. **使用通用模板** ⛔ - 必须基于项目实际情况
3. **推倒重来** ⛔ - 必须评估现有文档价值

```
❌ 不扫描 lib/ 就直接生成文档
❌ 复制其他项目的模板
❌ 不考虑现有基础就新建系统

✅ 必须先完成代码深度分析
✅ 必须基于现有组件生成文档
✅ 必须基于实际测试创建截图目标
```
