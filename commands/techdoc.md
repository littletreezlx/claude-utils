# 技术文档撰写

## 核心目标

根据用户指定的主题，生成两份职责分离的技术文档：**设计综述**（面向产品/管理层）和**实施手册**（面向开发团队）。文件名用中文。

## 输入方式

```bash
/techdoc "用户管理系统统一认证方案"
/techdoc $ARGUMENTS
```

---

## 文档 1: 设计综述 (Design Specification)

**受众**：产品经理、后端开发、管理层

**写作规则**：
- 禁止代码、类名、文件路径
- 图表使用 Mermaid 格式
- 聚焦"为什么"和"做什么"，不讨论"怎么做"
- 术语本地化：用业务词汇替换技术术语（"每小时入库的指标" > "Transaction with measurements"）

**内容结构**：
1. 核心价值与背景
2. 总体设计方案（含 Mermaid 流程图/时序图）
3. 关键决策对比（表格对比方案优劣）
4. 业务影响与风险

### §1 核心价值与背景 的内容顺序（关键）

读者读完 §1 必须能在 30 秒内回答：**"上线后大盘多出来什么？能回答什么业务问题？"**

因此 §1 的子节顺序必须是：

1. **§1.1 方案定位** — 一句话价值（把 X 从 Y 变成 Z）
2. **§1.2 核心指标** — 2-3 个决策级指标的表格，每行含"含义 + 能回答的业务问题"
3. **§1.3 大盘看板形态** — 能产出的 2-3 张看板，每张对应一种业务用法
4. **§1.4 现状痛点 / 为什么自建** — 铺垫类内容放最后

**反模式**：把"业务场景特性"、"现有系统约束"、"默认方案为什么不够"放 §1 开头。这些是论据不是价值，读完还不知道产出什么。

### 核心指标数量限制

§1.2 的核心指标**限 2-3 个**，每个都要是"能驱动业务决策的数字"。

辅助性指标（如总帧数作为样本基数、归一化分母）**降级为一句话**："同时附带一个辅助基数：X，用于 Y，不作为核心决策指标"。

**反模式**：并列列出 4-5 个指标让读者自己挑重点——读者不会挑，只会觉得"还行"然后忘掉。

### §3 关键决策对比 的写法

每个决策用"对比表 + 结论句"，不用辩护段落：

```markdown
### 3.1 <决策名词短语>

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| ... | ... | ... |

→ **结论：选 X**。<一句话理由>。
```

## 文档 2: 实施手册 (Implementation Guide)

**受众**：开发团队

**写作规则**：
- 代码优先，用代码解释逻辑
- 使用项目实际技术栈的规范写法
- 每个核心组件：代码块前说"设计取舍"，代码块后写"技术思考"
- 术语充分暴露：Transaction name、op、字段名等具体要写清，方便开发检索

**内容结构**：
1. 模块与文件结构（树状图）+ **命名规范小节**（Transaction name / op / tag / 数据库表名等全局查询入口）
2. 依赖配置（含版本选择的 rationale）
3. 核心接口定义（完整代码）+ 每个组件的"技术思考"段
4. 接入集成（改哪些已有文件、加哪些钩子）
5. 测试与验证（单测 + 联调清单 + 看板配置）
6. **性能预算与运行时开销**（每帧/每秒开销、内存、流量估算、风险项清单表）

### 每个核心组件的"技术思考"段（关键）

仅贴代码的实施手册价值不足——代码开发本来就看得懂。真正有价值的是**代码周围的思考**。每个核心组件（FrameCollector / XxxManager 等）的代码块后，必须追加一段"技术思考"，涵盖至少以下 5 类问题中的 3 类：

| 类别 | 示例问题 |
|------|---------|
| **并发与线程模型** | 为什么用 AtomicXxx 而不是普通变量？锁粒度为什么这么定？回调在哪个线程？ |
| **API 选型的坑** | 为什么用方法 A 而不是 B？有没有被废弃/不存在的字段要避开？（附反编译/源码验证路径） |
| **Edge case 与防御** | 重入、重复调用、冷启动、进程杀、配置变更下的行为 |
| **替代方案对比** | 手写 vs 官方库 / synchronized vs ConcurrentHashMap / 协程 vs WorkManager |
| **参数/阈值的依据** | 为什么是 100 而不是 50？为什么是 1h 而不是 30min？阈值的业界对齐来源？ |

### 风险项清单表（§6 必备）

性能预算章节必须有一张风险清单表：

```markdown
| 风险 | 触发条件 | 当前处理 | 升级路径 |
|------|---------|---------|---------|
| 进程崩溃丢数据 | Crash / LMK | 接受丢失 | 引入 MMKV 刷盘 |
| ... | ... | ... | ... |
```

每行 4 列。"当前处理"列要诚实写清楚（"接受丢失" / "不处理"也是有效回答），不要用"已优化"这种模糊措辞。

### 实施手册的 Pitfall 章节

如果踩过值得记录的坑（如某 API 看起来存在但实际不存在、某字段名拼写反直觉），单独成段落标注"这些错误别犯"，附反编译/源码/官方文档的一行引用作为证据。

## 输出格式

- 两份文档之间用 `---` 分隔
- 直接输出，不加开场白或过渡语
- 标题直接使用内容主题（如 `# 播放质量度量体系`），不要加"设计综述："、"实施手册："等前缀
- 输出到项目 `docs/<领域>/` 目录（如 `docs/performance/`），文件名英文：`<主题>-design.md` + `<主题>-implementation.md`

### 标题风格规则

所有二级以下标题（`##` `###`）用**断言式**或**名词短语**，**禁止"为什么 X"**：

| 反例 | 正例 |
|------|------|
| 为什么不用 MMKV 断电容灾 | 不引入断电容灾 |
| 为什么 tracesSampler 必须返回 1.0 | 幽灵事务强制采样（唯一不可省略项） |
| 如何保证线程安全 | 并发锁的职责分工 |
| 核心价值是什么 | 方案定位 |

**原理**：标题 = 结论。读者不想读"提问-回答"的辩论稿，想读"结论-论据"的决策档案。"为什么 X"把作者摆在辩护位置，弱化权威感。

## 约束

### 禁止元信息和交叉引用

每份文档是独立的，禁止：
- 受众说明（"本文档面向…"）
- 指向另一份文档的链接或引用（"实施细节见…"、"设计综述见…"）
- 关联文档列表（"关联文档：…"）
- 任何导读、导航、尾注性质的段落
- 文档到最后一个正文章节结束即止，不追加任何附加段落

### 禁止代码视角措辞

即使不写代码，也不要用暴露代码审查视角的措辞。禁止：
- "客户端代码中看不到 X"、"无 X 代码"、"无 X 痕迹"、"客户端可见度"
- "反编译确认"、"APK 反编译"、"反编译看到 / 看不到"

替代：用归属性肯定措辞——"X 在 Y 层实现"、"客户端不承担 X"、"X 由 Z 承载"。

例外：实施手册的 Pitfall 章节允许说"反编译验证 API 签名"等，因为那就是给开发看的证据链。

### 断言优先，避免防御性措辞

常识或架构事实可推断的结论用断言式陈述。禁止用"极可能 / 大概率 / 完全可以 / 疑在 / 应该是"等推测词掩盖不确定性；真不确定的地方明确写"未实测"、"未验证"，不要用模糊词蒙混。

### 推荐必须基于已验证事实或声明的假设

给方案推荐时不要拍脑袋假设前提（如"有自有 X 体系"、"团队已具备 Y 能力"）。每条推荐理由要么链到事实（实测数据、竞品对照、工程量估算），要么明确标注为假设（"假设 X，则建议 Y"），让读者能独立判断前提是否成立。

### 术语分层（设计综述 vs 实施手册）

同一概念在两份文档中的表达必须分层：

| 概念 | 设计综述措辞 | 实施手册措辞 |
|------|------------|------------|
| Sentry Transaction with op | "每小时入库的指标记录" | "`Sentry.startTransaction(name, "apm.runtime")`" |
| 页面切换差值法 | "按页面归类的帧数据" | "`enterSnapshot` + `settleInternal`" |
| 1 小时聚合 | "每小时上报一次" | "`ApmScheduler` 协程 `delay(INTERVAL_MS)`" |

**违反信号**：设计综述里出现反引号代码字体、类名、方法名、配置项名 → 应该下移到实施手册。

### 迭代修改原则

用户反馈时不要推翻重写。按反馈点定位：
- 标题/段落顺序问题 → 重排目录，保留正文
- 内容深度不够 → 逐节追加"技术思考"段
- 术语不当 → 跨文档分层迁移，不删除信息
- 指标/维度取舍 → 降级而非删除（"辅助基数，不作为核心决策指标"）

### 更多撰写卫生

竞品稿 vs 决策稿分工、自谦措辞、删段重编号脚本、上飞书前 lint 等，详见 `~/.claude/guides/tech-report-hygiene.md`。

### 文件名

输出文件名必须使用英文，格式：`<主题>-design.md` / `<主题>-implementation.md`。禁止中文文件名。

---

## 执行参数

`$ARGUMENTS` - 文档主题或需求描述。如果用户在主题后追加指令（如"放到 docs/xxx 目录"、"全局采样率是 5%"），把这些作为上下文约束纳入文档，不要忽略。
