# 项目级重构（DAG 编排）

> 系统性重构项目架构，支持并行执行和断点续传

## 使用方式

```bash
/comprehensive-health-check              # 可选：先诊断
/refactor-project                        # 生成重构任务
batchcc task-refactor-project            # 执行
```

> **格式规范**：
> - @templates/workflow/DAG_FORMAT.md - DAG 统一规范（**必须遵循**）
> - @templates/workflow/REFACTOR_TASK_TEMPLATE.md - 重构任务模板

---

## 自主执行原则

> 核心：自主分析 → 自主决策 → 直接执行 → 记录理由（不询问用户）

---

## AI 可维护性优先（核心原则）

> AI 不嫌文件大，嫌文件多且分散。理解 500 行内聚代码 < 跳 5 个 60 行文件组装逻辑。

### 重构判断标准

#### 架构层（优先级高，先看这些）

| 判断维度 | 生成任务 | 不生成任务 |
|---------|----------|-----------|
| 分层架构违规？ | 跨层直接调用、绕过抽象层 → 迁移 | 调用链路符合架构约定 → 不动 |
| 接口契约不一致？ | 同类模块对外接口风格不统一 → 统一 | 已统一 → 不动 |
| 依赖方向违规？ | 底层模块反向依赖上层 → 解耦 | 依赖方向正确 → 不动 |
| 技术债务集中度？ | 同一问题分散 N 个文件 → 统一迁移 | 个别文件 → 局部修复 |

#### 代码层（架构层无问题后再看）

| 判断维度 | 生成任务 | 不生成任务 |
|---------|----------|-----------|
| 职责是否单一？ | 混乱 → 拆分 | 单一 → 保持 |
| 是否会被复用？ | 会 → 提取组件 | 仅一处用 → 保持内聚 |
| 有循环依赖？ | 有 → 消除 | 无 → 不动 |
| 代码重复？ | 有 → 统一 | 无 → 不动 |
| 目录内聚性？ | 目录内文件松散无关 → 重组 | 文件紧密关联 → 保持 |
| **文件行数** | **不作为判断依据** | - |

### 判断流程

```
读取文件 → 理解职责 →
├── 职责单一且内聚？→ 保持原状（不管行数）
├── 职责混乱？→ 按职责拆分（不是按行数切）
├── 有可复用的部分？→ 提取到共享位置
└── 目录内文件松散？→ 按功能归类到子目录
```

## 执行策略

### 第一步：充分探索项目

**规划质量决定执行效率**：
- 阅读核心文档（README、CLAUDE.md、docs/ROADMAP.md）
- 浏览项目结构，理解模块划分和依赖关系
- **运行 `./scripts/quality/invariants.sh`，逐条审视 WARNING/VIOLATION** — 这些是 `scripts/quality/invariants.sh` 中定义的架构不变量检查结果，代表已识别但未修复的架构问题，优先级高于新发现的局部问题
- **对比上一轮重构的"明确不做"列表** — 如果同一项连续 2 轮被排除，本轮必须纳入或给出升级理由
- 查找 TODO/FIXME 注释（真实痛点）
- 按优先级识别：**分层违规 > 架构债务** > 职责混乱 > 循环依赖 > 代码重复

### 第二步：生成任务编排文件

**必须生成两个产出**：
1. **入口文件** `task-refactor-project`（项目根目录）
2. **任务细节目录** `.refactor-tasks/`

### 第三步：各阶段设计

| 阶段 | 内容 | 执行模式 |
|------|------|---------|
| Stage 1 | 各模块独立重构：职责拆分、接口优化、**目录内聚性整理** | **并行** max_workers=4 |
| Stage 2 | 跨模块整合：消除重复轮子、提取共享基础设施 | 串行 |
| Stage 3 | 全量测试 + 集成验证 | 串行 |
| Stage 4 | 架构文档、功能映射、ADR | 并行 |
| Stage 5 | 全局审视 + /todo-write 收尾 | 串行 |

## 重要约束

1. **架构优先于代码** - 分层违规、依赖方向错误比局部代码质量问题更重要
2. **治本优先于治标** - 37 个文件的统一迁移 > 3 个文件的行数优化
3. **职责优先于行数** - 职责单一的大文件 > 职责混乱的多个小文件
4. **内聚优先于复用** - 不复用的代码提取反而增加复杂度
5. **测试驱动** - 每阶段测试验证，不过不进下一阶段
6. **不要过度设计** - 解决现有问题，不为"可能的未来"设计

## 严格禁止

1. **回避核心债务** - 如果 `invariants.sh` 的 WARNING 已存在 2 轮以上，禁止再次排除到"明确不做"
2. **以行数为目标** - `行数 ≤ 200` 不是重构目标
3. **机械拆分** - 只在职责混乱时拆分
4. **为复用而复用** - 提取不复用的组件是过度设计
5. **用低风险任务凑工作量** - 类型注解补全、linter 可自动修复的问题不值得占 Stage
