# PRD 转客户端需求

## 目标
从全栈 PRD 中提取并澄清客户端相关需求，输出客户端开发视角的需求文档。

## 适用场景
- PM 给了一份涵盖前后端的总体 PRD
- PRD 对客户端具体要做什么、改哪里是模糊的
- 需要一份客户端视角的需求澄清

## 与其他命令的区别
| 命令 | 输入 | 核心动作 | 输出 |
|------|------|----------|------|
| `/feat-discuss` | 模糊想法 | 讨论澄清 | 功能方案 |
| `/prd-to-doc` | 全栈 PRD | 提取+澄清客户端部分 | 客户端需求文档 |
| `/speckit.specify` | 明确需求 | 细化规格 | 详细功能规格 |

## 执行方式

### 1. PRD 分析
- 阅读总体 PRD，识别客户端相关需求
- 区分：纯客户端功能 / 依赖后端的功能 / 纯后端功能（忽略）
- 标记模糊或缺失的细节

### 2. 澄清讨论（1-2 轮）
针对模糊点提问，典型问题：
- 交互流程是什么？
- 数据从哪来？（本地 / 后端 API）
- 异常情况如何处理？
- 有设计稿吗？

### 3. 输出客户端需求文档

**核心要素**（按需取舍）：
- 需求概述（一句话核心目标）
- 功能清单（区分 UI / 逻辑 / 数据，标注后端依赖）
- 后端依赖（需要的 API 列表）
- 待澄清项（需要找 PM 确认的问题，如有）
- 实施建议（推荐的开发顺序）

## 文件生成
澄清完成后，生成本地文档：
- **路径**：`docs/client-requirements/` 或项目约定的文档目录
- **命名**：`{功能名}-client-req.md`
- **内容**：上述客户端需求文档格式

## 约束
- 聚焦客户端视角，不深入后端实现细节
- 发现 PRD 有重大缺陷时，先反馈给 PM
- 必须生成本地文档，方便后续开发和任务拆分

## 执行参数
`$ARGUMENTS` - PRD 文档路径或内容
