# 项目级 CLAUDE.md 生成与维护

## 核心目标

分析项目特征，生成或更新**项目级** CLAUDE.md，为 Agent 提供精准的执行契约。

## 使用方式

- `/claudemd` — 自动检测：无 CLAUDE.md 则生成，有则增量更新
- `/claudemd init` — 强制从零生成（覆盖现有）
- `/claudemd audit` — 深度分析现有 CLAUDE.md，评估质量并优化
- `/claudemd $ARGUMENTS` — 针对特定变更定向更新（如"新增了 WebSocket 模块"）

---

## 执行策略

### Phase 1: 项目探索

充分理解项目再动手。重点收集：

- 项目配置文件 → 技术栈和版本
- 目录结构 → 模块划分
- 已有文档（README、docs/）→ 架构意图
- 测试配置和脚本 → 测试命令
- CI/CD 配置 → 构建/部署命令
- git log 最近 20 条 → 开发风格

### Phase 2: 模式识别（Agent 友好性扫描）

**技术模式**：状态管理、网络请求、错误处理、依赖注入、命名约定 — 是否统一？

**反模式扫描**：

- `TODO`/`FIXME`/`HACK` 注释 → 已知技术债务
- 类型安全问题（`dynamic`/`any`/空 catch）→ 写入禁止事项
- 硬编码配置/密钥

**模式冲突检测**（同一件事多种做法）：

- 发现冲突时，判断哪个是"正确模式"，写入 CLAUDE.md 作为决策

### Phase 3: 生成/更新

**增量更新模式**（已有 CLAUDE.md）：diff 对比，只改变化部分，保留用户手动添加的规则。

**内容结构**（只写项目特异性信息）：

**必写**：

- 项目概述（一句话）
- 技术栈（框架版本、关键依赖）
- 执行命令（构建、测试、lint、格式化，含具体参数）
- 架构决策（状态管理、网络层、存储等统一方案）
- 禁止事项（项目特有的反模式红线）

**按需写**：

- 模块说明（如果 docs/ 不够详细）
- 已知技术债务（从扫描提炼）
- 环境配置（特殊的环境变量或配置）

### Phase 4: 自我验证

- 每条规则是否有代码证据支撑？无证据 → 删除
- 命令是否可直接复制执行？
- 删掉某行后 Agent 行为是否会改变？不会 → 删掉

## Audit 模式

当参数为 `audit` 时，跳过 Phase 1-3，改为对现有 CLAUDE.md 执行深度分析。

### 基础评估

**1. 体量与信息密度**

- 统计总行数和有效指令数（含 If/Then 结构的规则）
- 计算信息密度：有效指令数 / 总行数
- 超过 200 行时建议精简或拆分，但不以行数为唯一标准

**2. 删除测试（逐段）**

- 对每个段落/规则问："删掉后 Claude 行为是否改变？"
- 🔴 不会改变 → 标记为冗余，建议删除
- 🟡 Claude 大概率已知 → 标记为低价值，建议删除或简化
- 🟢 项目特异性强 → 保留

**3. 职责越界检测**

- 代码风格规则是否应交给 linter/formatter？→ 建议迁移到工具配置
- 通用编程常识是否写入了 CLAUDE.md？→ 建议删除
- 详细文档是否内联？→ 建议用 `@path` 引用替代

**4. 时效性验证**

- 引用的文件路径是否实际存在？（`Glob` 验证）
- 引用的命令是否可执行？（`Bash` 抽样验证）
- 提及的技术栈版本是否与当前配置一致？

### 深层分析

**5. 触发器明确度（Trigger Clarity）**

- 扫描所有指令，检查是否包含明确的触发条件
- 模糊指令（"平时要多…"、"尽量保持…"、"适当运用…"）→ 🟡 建议重构为 "当 [事件] 发生时，执行 [动作]" 格式
- 有明确 If/When/Then 结构的指令 → 🟢 高可执行性

**6. 指令冲突检测（Deadlock Detection）**

- 识别可能互相矛盾的规则对
- 示例：规则 A "修 Bug 时不要修改非核心文件" vs 规则 B "发现坏味道随时清理"
- 发现冲突 → 要求用户设定优先级（"Rule X overrides Rule Y when [condition]"）

**7. ROI 评估**

- 对每条规则评估：`可执行性 (0-1) × 违反损害度 (1-5)`
- 高 ROI（可执行 + 高损害）→ 🟢 核心规则，必须保留
- 低 ROI（模糊 + 低损害）→ 🔴 下放给自动化工具或直接删除

### 输出格式

```markdown
## CLAUDE.md 审计报告

### 概览
- 总行数：XX | 有效指令数：XX | 信息密度：XX%
- 评级：🟢 精炼 / 🟡 可优化 / 🔴 需重构

### 发现

#### 🔴 建议删除（X 项）
- [行号范围] 原因 | ROI: X

#### 🟡 建议精简（X 项）
- [行号范围] 原因 → 建议改为 | 触发器: 模糊→明确

#### 🔵 建议外移（X 项）
- [行号范围] → 移至 `.claude/rules/xxx.md` 或 `@docs/xxx`

#### ⚠️ 指令冲突（X 对）
- 规则 A [行号] vs 规则 B [行号] → 建议优先级

#### 🟢 高价值保留（X 项）
- [行号范围] 为什么有价值 | ROI: X

### 优化方案
（直接给出优化后的 CLAUDE.md，用户确认后写入）
```

### Audit 后的操作

- 报告仅展示，不自动修改 CLAUDE.md
- 用户确认优化方案后，再执行写入
- 如果涉及拆分到 `.claude/rules/`，一并创建对应文件

---

## 质量标准

- **特异性**：不写全局 CLAUDE.md 已覆盖的通用规则
- **可执行性**：每条规则包含具体的触发条件和动作（"当 X 时，做 Y"）
- **准确性**：每条规则与当前代码一致
- **信息密度**：每行都应通过"删除测试"，追求指令密度而非行数限制

## 约束

- 不写 docs/ 体系已覆盖的详细架构文档
- 增量更新时不破坏用户手动添加的规则
- 不推测未验证的模式（扫描不到证据就不写）
- audit 模式不自动修改文件，必须用户确认
