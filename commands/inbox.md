# Inbox 自动归档

## 核心目标
扫描 `99-系统/01-Inbox/` 中的待整理文档，根据内容语义自动路由到知识库的正确目录位置。

## 输入格式
- `/inbox` — 处理 Inbox 中所有文件
- `/inbox <文件名>` — 只处理指定文件

## 知识库根目录
`~/LittleTree_Projects/MyForest/`

## 执行流程

### Step 1: 扫描 Inbox
列出 `99-系统/01-Inbox/` 下所有文件（排除 `.gitkeep` 等系统文件）。
如果 Inbox 为空，告知用户并结束。

### Step 2: 逐文件分析
对每个文件执行以下分析：

1. **读取内容**：读取文件前 2000 字符，获取主题和关键词
2. **查阅路由规则**：读取 `CLAUDE.md` 第 6 节「目录导航语义（AI 路由配置）」
3. **L1 定位**：根据内容主题匹配最佳 L1 领域
4. **L2 定位**：在 L1 内匹配最佳 L2 子领域
5. **L3 定位**：在 L2 内匹配最佳 L3 分类。如无匹配，放在 L2 根目录
6. **查重**：在目标目录下搜索是否已有相关主题的文档
   - **有高度相关文档**：建议合并（追加到已有文档），而非新建
   - **无**：直接移动

### Step 3: 跨领域判断
当文档涉及多个领域时，应用核心原则：
- **按目标/场景归文件夹，按工具/手段打标签和双链**
- 示例："用 AI 做投资分析" → 文件放 `06-投资/`，在文档 frontmatter 或正文中添加 `#AI` 标签

### Step 4: 置信度分级与执行

#### 高置信度（明确属于某个目录）
直接执行移动，报告结果：
```
✅ [移动] 文件名.md → 01-技术/03-移动端开发/Android原生/
```

#### 中置信度（2-3个候选目录）
列出候选方案，等待用户选择：
```
📋 文件名.md — 检测到多个可能归属：
  1. 01-技术/01-计算机基础/网络协议/ (网络相关内容)
  2. 03-工作/当前工作/技术文档/ (工作项目相关)
请选择 (1/2)，或输入自定义路径：
```

#### 低置信度（无法确定）
保留在 Inbox，标记原因：
```
⏸️ 文件名.md — 无法确定归属，保留在 Inbox
   原因：内容过短/主题模糊/跨领域无法判断
```

### Step 5: 文件名规范化
移动时检查文件名是否需要调整：
- 确保中文或关键英文术语命名，简洁明了
- 如果目标 L3 目录下使用了前缀命名法（如 `性能优化-APM.md`），新文件也应遵循
- 不改变文档内容，只调整文件名和位置

### Step 6: 汇总报告
处理完所有文件后，输出汇总：
```
📊 Inbox 归档完成
  ✅ 已归档: N 篇
  📋 待确认: N 篇
  ⏸️ 已跳过: N 篇
```

## 目录层级规范

遵循知识库的 **「3层骨架 + 4层容忍 + 5层熔断」** 原则：

| 层级 | 规则 |
|------|------|
| L1-L3 | **骨架层**，直接使用已有目录 |
| L4 | **容忍层**，仅当该分类下 ≥5 篇文档时才建子目录；<5 篇用前缀命名法 |
| L5+ | **熔断**，禁止创建 |

**重要**：归档时优先使用已有目录，不轻易创建新目录。新建 L3 目录前必须确认无语义重叠的已有目录。

## 约束条件
- **不修改文档内容**：只移动文件，不改写正文（除非需要添加标签）
- **不破坏已有结构**：不重命名或移动已在正确位置的文件
- **异物隔离**：非知识文档（代码、二进制等）不归档到知识区，保留在 Inbox 并提示用户
- **新建目录前检查**：① 层级 ≤ 4？ ② 预期文档数 ≥ 5？ ③ 与已有目录无语义重叠？
- **禁止运行代码**：所有操作通过文件系统工具完成
- **中置信度必须确认**：不确定时等用户确认，不强行归档
