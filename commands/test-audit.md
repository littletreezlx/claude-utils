# 测试基础设施审计与修复

> 审计测试基础设施（能不能跑），直接修复缺失的配置和脚本

```bash
/test-audit                    # 审计并修复当前项目
/test-audit $ARGUMENTS         # 指定关注点（unit/integration/e2e）
```

## 核心原则

1. **审计即修复** — 发现问题直接解决，不询问
2. **只动基础设施** — 不修改业务代码和测试用例
3. **技术栈适配** — 根据项目类型选择方案
4. **降低决策熵** — 测试模式应全项目统一，Agent 看到一种做法就够了

---

## 检查项

- [ ] 测试命令配置完整（unit/integration/e2e/watch/coverage）
- [ ] E2E 脚本具备：进程清理、依赖服务管理、报告生成
- [ ] 测试可正常运行（执行验证，不只是检查配置）

**缺失时**：根据项目实际情况直接生成适配的脚本和配置。

> **注意**：CI/CD 配置不在审计范围内（个人项目通常不需要）。

## 完成后

- 在对话中输出审计结果（项目类型、框架、各类型测试状态、已修复项）
- 检查测试模式一致性（见下方），将结论更新到 `CLAUDE.md` 测试命令部分

### 测试模式一致性检查（Agent 决策熵）

**同一件事是否有多种做法？** 抽查检测：

| 检查项 | 混用示例 | 统一方向 |
|--------|---------|---------|
| Mock 方式 | jest.mock / manual mock / dependency injection 混用 | 选主流方式，记录到 CLAUDE.md |
| 断言风格 | expect/assert/should 混用 | 统一到一种 |
| 测试数据 | 硬编码 JSON / factory / fixture 混用 | 统一到 factory 或 fixture |
| 异步测试 | callback / promise / async-await 混用 | 统一到 async-await |

发现混用 → 记录到输出报告，建议统一方向（不自动重写已有测试）。

## 禁止事项

1. 不使用模板文件 — 根据项目实际情况生成
2. 不生成报告文件 — 只在对话中输出
3. 不修改业务代码 — 只处理测试基础设施
