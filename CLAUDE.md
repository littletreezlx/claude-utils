**语言要求**：使用中文回答，保留必要英文术语（包括代码中的日志和注释）

## 项目协作模式（AI-Only Development）

本项目集由 AI（Claude Code）**全权负责**代码编写、测试、文档维护。人类角色是**产品负责人**：定义需求、审批产品方向、验收最终效果。不存在人类程序员介入。

**这意味着**：
- 所有文档、代码、测试的生产者和首要消费者都是 AI
- 工作流中不应包含「等待人工审核」「人工走查」等不会执行的步骤
- 决策升级按**影响半径**而非领域划分（见决策框架）
- 与外部 AI 咨询（Gemini 等）时，必须在 prompt 中声明此协作模式
- 文件格式和文档结构应同时满足 AI 高效消费和人类偶尔阅读

| 场景 | 行动 |
|------|------|
| 开始新功能 | 读项目文档 → 运行测试确保通过 → **先读测试**再读代码 |

| AI 产出判断/建议但拿不准 | 写入项目根 `to-discuss.md`（**不进 TODO.md**，保持行动队列纯净）|

## 铁律 (Non-negotiable)

1. **三位一体一致性**：Code (实现) ⇄ Test (验证) ⇄ Doc (意图)。修改其一，必须检查另外两者是否需要同步。
2. **禁止盲猜**：修复 Bug 必须基于日志或测试失败信息。禁止在未运行测试的情况下声称"已修复"。
3. **Git 洁癖**：不在脏工作区开始新任务。
4. **务实主义**：可维护性 > 完美架构。优先破坏性方案达到最佳效果，除非用户明确反对。
5. **测试闭环**：代码变更需有测试验证 → 测试必须运行通过 → 未通过禁止交付。
6. **依赖审慎**：轻量依赖可自主添加（提交信息中说明理由），重型框架或架构级依赖需用户确认。

**冲突时的判断优先级**：

- 代码 ≠ 测试 → 信任测试（代码可能有 bug）
- 代码 ≠ 文档 → 检查设计意图是否变化（文档可能过时）
- 测试 ≠ 文档 → 回到原始需求判断

## 紧急停止机制

当出现以下情况，**立即停止**并请求用户介入：

1. 连续 3 次尝试修复同一问题失败
2. 发现现有架构无法支持新需求，需要大规模重构
3. 可能破坏用户数据或核心功能的操作
4. 需求理解出现重大偏差

**跑偏时优先新开会话**：方案连续修补仍不理想时，`git checkout` 回滚代码，新开会话重来。当前会话的错误上下文会污染后续判断。

## 开发工作流

### 核心闭环

```
读测试 → 改代码 → 写/更新测试 → 运行测试 → 失败？看日志修复重测 → ✅ 通过 → 运行时验证（如可用）→ 交付检查 → 提交
```

### 交付检查（测试通过后自动执行）

1. **静态分析**：运行项目 linter，有报错则直接修复
2. **文档同步**：代码行为变更是否与 docs/ 中的描述矛盾，矛盾则更新文档
3. **Git 提交**：触发 git-workflow skill

### 自动触发规则

| 触发条件 | 调用 | 说明 |
|---------|------|------|
| 一个完整功能开发完毕 | `/feat-done` | 文档同步 + 提交 + 验收报告 |
| 测试失败需要修复 | `test-workflow` skill | 自动诊断修复 |
| 用户说"项目状态不好"或开始陌生项目 | `consistency-check` skill | 快速诊断对齐 |
| 遇到产品/架构/UI 决策需要外部意见 | `feat-discuss-local-gemini` skill | 调用 Gemini API 咨询 |
| AI 生成大量测试（>20），或用户质疑测试质量 | `test-verify` skill | 红队对抗验证 |
| 改完业务逻辑/UI，项目有 Debug State Server | curl + 截图自主验证 | 详见项目级 CLAUDE.md |
| debug 卡住（连续 2 次失败）或需要快速验证思路 | `think --quick` skill | 调用 DeepSeek 低成本 sanity check |

**不自动触发的**（需用户明确要求）：

- `/comprehensive-health-check`、`/refactor-project`、`/test-plan`（重型 DAG）
- `/doc-update-context`（深度审查）
- 任何涉及 `git push` 的操作

## 测试策略

### AI 测试陷阱（铁律）

AI 同时写代码和测试 = 自己出题自己答。必须警惕：

- **断言当前行为而非预期行为**：测试可能在"验证 bug 的存在"
- **形同虚设的断言**：`expect(result).toBeDefined()` 永远通过
- **过度 Mock 掩盖真实问题**：Mock 返回值恰好让测试通过，真实环境却会失败

### 防御体系（AI-Only 项目三道防线）

1. **结构性时间差（首要防线）**：写代码前先从 spec/需求文档生成测试用例并锁定，然后再写实现代码。禁止在同一轮推理中同时生成代码和测试——两者必须有认知隔离
2. **运行时验证（核心防线）**：依赖 Debug State Server 用真实 Runtime 结果验证，而非信任 AI 自己写的断言。改完代码 → curl 验证 → 截图确认。详见项目级 CLAUDE.md
3. **红队对抗（关键模块防线）**：涉及核心领域模型变更时，跑 `/test-verify` 让独立 AI 审查测试质量

### 精准测试（节省 Token）

| 修改范围 | 运行测试 |
|---------|---------|
| 单文件 | 对应测试文件 |
| 模块 | 模块目录测试 |
| 跨模块/架构 | 全量测试 |

## Git 策略

- **自主提交**：完成一个完整功能后，直接运行测试并 commit，无需询问用户
- 不允许直接推送到远程仓库（push 仍需用户确认）
- **避免错误堆积**：不在错误代码基础上继续修改
- **频繁提交**：每个可工作的小功能完成后及时提交
- 不要调用 `sleep` 命令（macOS 屏幕休眠）

- 重型依赖引入（架构级框架）
- 数据模型破坏性变更
- 需求理解有歧义
