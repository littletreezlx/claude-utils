# 2026-04-24-04 /think skill 0.6.0 — 核心/进阶分层 + 默认模式降级 + 评审四点修补

**类型**:Skill 结构瘦身(层积式复杂度的第一次减法)+ 评审发现的语义修补
**影响半径**:`~/.claude/skills/think/SKILL.md` + 新建 `~/.claude/skills/think/ADVANCED.md`
**版本**:think 0.5.2 → 0.6.0

## 遇到什么问题

Founder 在评审 think skill 设计时提出:"是否设计的过于复杂?"

Claude Code 核对后判定:**复杂度处于"刚好够用略偏重"区间,不是过度但正在滑向过度**。同时第一轮评审发现 1 个真 bug + 4 个设计张力。四个危险信号:

### 信号 1:版本节奏只加不减

0.4 → 0.5.0 → 0.5.1 → 0.5.2 都在 2026-04-18 一天内完成,每次都在加字段/加校验:

- 0.4.0:术语中文化 + 自包含规则 + 会话类型分叉
- 0.5.0:引入 DUAL 模式 + 2 个新字段 + 判优机制
- 0.5.1:核心主张长度约束 + 事前验尸反证降级 + Digest/升级模板分工规则 + 选项互斥性约束
- 0.5.2:综合推荐字段 4 条强制要求

**没有一次做减法**。典型的层积式膨胀——没人敢删已加的东西,怕踩回旧坑。

### 信号 2:元规则出现(规则在解释自己的规则)

3.3.3 节"Digest 与升级模板的分工规则"(0.5.1 新增)本身就是在修 0.5.0 Dual 模式 + 升级场景下的双重展开问题。Dual 引入的复杂度已经溢出过一次,需要额外一层元规则打补丁。

### 信号 3:装饰性条款累积

- **6 维度判优表**:Claude Code 作为 judge 真会按 6 维逐一打分吗?现实大概只用 1-2 维
- **升级模板"事实前提"在交互模式被标"可省略或精简"**:那为什么还强制进模板骨架
- **多轮追问的滑动窗口+已锁定共识+每轮 4 子块协议**:很精致,但多轮占总调用量比例未知,主路径可能在为低频场景买单

### 信号 4:430 行占用 AI 上下文

Claude Code 每次执行 skill 都要消化全文 430 行。AI 的注意力被结构条款稀释,反而让真正的思考空间变小——skill 的初衷是"提高咨询质量",但形式约束过密会让 Claude 把认知花在"满足 Digest 7 字段校验"。

### 元层反讽

这个 skill 对抗的是 **AI 的仪式化倾向**——走流程、填字段、不真思考。但 skill 自己用"更多结构化约束"去对抗,本身在制造更多可仪式化的形式。临界点:约束密度高到某个程度后,填字段的形式压力会超过思考压力。当前未到,0.5.x 节奏显示正在逼近。

### 评审第一轮发现的 4 个设计张力

除复杂度之外,评审还暴露出:

1. **真 bug:L293 引用未定义的 "P<x>"/"P 级排序" 概念**——违反自包含规则。来源是 2026-04-18-03 决策记录中的真实案例(flametree_ai 音频架构讨论事前验尸按 P1-P5 列出),但 P 级是当时 Claude 的产出格式,不是 skill 强制约定。
2. **事前验尸缺少质量下限**:"写不出答案说明不成熟" 对空洞版("最大风险是用户不喜欢")没有约束力,仪式齐全但推进无效。
3. **Dual 判优缺少反偏好校验**:Claude Code 同时是提问者+裁判,判优维度表里没有"赢家与我原始倾向一致时补验"的 confirmation bias 防御。
4. **触发条件 1 "跨项目影响" 边界不清**:"影响 ≥2 个项目"的"影响"是直接代码改动还是"思路会被借鉴"?过宽会导致每次全局 skill 改动都强制咨询。

## 决定什么

### 1. 默认模式 DUAL → Solo,由话题性质选 gemini/gpt

**旧默认**:无 flag 调用 think.mjs 自动走 DUAL(Gemini + GPT-5.4 并发,2x token)

**新默认**:Claude Code 按话题性质显式选择:
- **偏逻辑推演 / 战略分析 / 算法设计** → `--solo gpt`
- **需多模态(图片) / 横向联想 / 开放探索** → `--solo gemini`
- **无明确倾向** → `--solo gemini`(兜底,多模态更通用)
- **真有策略分歧 / 影响极广 / 需双方对抗** → `--dual`(Claude Code 明确判断或用户显式请求)

**理由**:
- 方法论/策略/哲学/元认知是 think 的核心定位,纯逻辑占多数,GPT-5.4 通常够用
- DUAL 的 2x 成本只在真正策略分歧时有回报,大部分首轮是信息不对称而非立场对立
- 让 Claude Code 在首轮做一次"话题性质分类"比盲目 DUAL 更负责

### 2. 主/进阶分层:SKILL.md 瘦身 + 新建 ADVANCED.md

**SKILL.md(核心层)保留**:
- 用途 + 与 think-gem-project 分工
- 3 种模式的选择逻辑
- 3 条触发硬信号
- Step 1 输入契约
- Step 2 think.mjs 基础调用 + 图片支持
- 首轮 Prompt 格式(6 段)
- Step 3.1/3.2 Digest 核心(Solo 5 字段 + 校验 + 反证降级)
- Step 3.3 升级路由(命中 + 会话分叉 + 升级模板骨架)
- Step 3.5 反仪式性自检
- Prompt 组装原则 + 约束 + 变更历史

**ADVANCED.md(进阶层)下沉**:
- Dual 模式完整细节(Digest 7 字段、判优机制、6 维度表、反偏好校验)
- 多轮追问 Prompt 格式(滑动窗口 + 已锁定共识 + 两方上下文)
- Digest 与升级模板的分工规则(元规则)
- 升级模板选项互斥性约束细则
- 综合推荐字段 4 条强制要求完整表述

**引用方式**:SKILL.md 在相关位置注明"若使用 `--dual` / 多轮追问 / 升级路径,详见 `skills/think/ADVANCED.md`",按需加载。

### 3. 修 L293 真 bug:去掉 P<x> 引用

**旧**:`[立场依据]` 必须显式标注"事前验尸 P<x> 被纠正,原排序置信度降级"

**新**:`[立场依据]` 必须显式标注"事前验尸包含被证伪的事实项,原判断置信度降级"

用"事实项"作为中性词覆盖单机制/多机制排序/等同各种事前验尸格式,不再依赖未定义的 P 级概念。

### 4. 事前验尸新增质量下限

首轮 Prompt 的"事前验尸"字段说明新增一条:

> **要求**:至少点名一个**可证伪的前提假设**——指向具体的数据/机制/前提(如"假设 GPT 首轮延迟 ≤3s"、"假设用户能识别色板差异"),而不是泛化情绪(如"用户不喜欢"、"技术栈不适合")。列不出可证伪假设 = 决策不成熟,停下来先想,不要 call think。

"可证伪"这个标准比"想出答案"更硬——它要求 Claude Code 指名**具体会在什么条件下被推翻**,而不是模糊担忧。

### 5. Dual 判优新增反偏好校验

ADVANCED.md 的 Dual 判优维度表新增一条"反偏好校验",且标为**必跑**:

> **反偏好校验(必跑)**:判优前先问"赢家是否与我原始倾向一致?"。若一致,`[模型判优]` 字段后补一句"反向情况下我会选谁、为什么这次不选"——强制 Claude Code 把 confirmation bias 显性化。若理由只是"另一方说得没这方好"而不能具体说出哪条论据更弱,判优结果可疑,重新权衡。

这是对 Claude Code "提问者+裁判"双重身份的结构性修补。

### 6. 触发条件 1 "跨项目影响" 边界收紧

**旧**:跨项目影响:改动会影响 ≥2 个项目的共享配置 / 工作流 / 代码

**新**:跨项目影响:**当前改动直接修改** ≥2 个项目可读取的文件(共享 skill / 全局 CLAUDE.md / 跨项目 doc / shared 工具脚本)。仅"思路会被借鉴"、"未来可能影响"不算——这些属于触发条件 2(持久化到规则)或 3(元认知指令)的覆盖范围。

去掉"影响"的软语义,锁死到"直接修改文件"的可机械校验事实。

### 7. 版本号 0.5.2 → 0.6.0

行为级变化(默认模式切换)+ 结构级变化(分层) + 语义修补(4 个张力)三叠加,按 semver 升 minor。变更历史首次出现 "+X / -Y / 修 Z" 的混合条目,标志层积式演进转向平衡演进。

## 为什么不选其他方案

### 方案 A:彻底删掉 Dual 模式

更激进——Dual 及其整套机制直接移除,只留 Solo + Quick。

**不选原因**:
- 2026-04-18 的三次决策记录积累了 Dual 的真实调优经验,机制本身是对的
- "真策略分歧 + 需双方对抗"场景确实存在,彻底删了日后还要重建
- 本次目标是"让复杂度按需出现"而非"移除复杂度",Dual 作为显式可选路径保留即可

### 方案 B:主 SKILL.md 内部分段,不拆文件

用"核心规则 / 进阶规则"两大段组织,仍在一个文件里。

**不选原因**:
- Claude Code 的上下文占用跟文件长度直接相关,单文件 430 行 vs 280+180 拆分,后者主路径真正瘦了 150 行
- 拆文件能显式表达"默认只读核心,进阶按需加载"的语义,比段落分隔更硬
- ADVANCED.md 独立演进时不污染主路径

### 方案 C:分批做(先分层,下一版再修 4 张力)

Claude Code 初版方案。

**不选原因**:
- 下一版只剩加法(加质量下限、加反偏好校验、收紧边界),没有减法陪跑,那时才真正退化为层积式
- 评审已经暴露清晰的修补点,没理由推迟
- 合并到 0.6.0 能让变更历史体现 "+X / -Y / 修 Z" 的平衡演进,这是新纪律的第一条示范
