## 规则

做 Android POS 竞品分析时,**禁止把"APK 反编译看不到某能力"等同于"竞品不提供该能力"**。必须先区分厂商的硬件控制权:

- **纯软件厂商**(如 qm/企迈):软件装在第三方硬件上 → 所有能力必须在 APK 内实现 → APK 反编译结论**可靠**
- **软硬一体厂商**(如 mt/美团、kry/客如云):自研 POS 机 + 定制 ROM → 能力可下沉到**系统服务 / ROM 层 / 厂商运维后台** → APK 反编译**有系统性盲区**

对软硬一体厂商的分析中,"APK 内无实现"只能推出"不在业务 APK 层",**不能推出**"竞品不提供该能力"或"这是产品取舍"。

## Why

2026-04-17 在 `docs/远程协助-对比.md` 中把 mt 总结为"根本不做远程桌面 / 认为屏幕级远程桌面不是必要的",被 Founder 纠正:mt 是软硬一体,远程协助完全可以做在系统/ROM 层,根本不需要出现在业务 APK 里。把 APK 反编译结论(事实)外推成了产品哲学判断(错误推论),三家"技术路线差异"被误读为"产品哲学差异",掩盖了真正的根本差异 —— **硬件控制权决定技术路径**。

## How to apply

- 写竞品 APK 反编译文档时,凡是结论依赖"APK 里没找到 X"的,**必须**先声明"以下结论仅基于 APK 反编译可见部分",再下结论
- 能力矩阵中对软硬一体厂商的"缺失项"标记 **❓ APK 不可见** 而非 ❌,并在表前说明
- 对比表的"方案类型/功能定位"维度,标题冠以"APK 层"前缀,避免和"整体能力"混淆
- 写"一句话总结"时,对软硬一体厂商必须点明"APK 可见部分 vs 可能下沉到系统层",禁止用"不提供 / 不支持 / 不做"这类绝对化表达
- 识别硬件控制权的线索:自研 POS 机品牌、定制 ROM、厂商后台管理系统的存在 → 都提示能力可能下沉
