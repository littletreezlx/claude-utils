## 模式名称：实现即目的（Means-as-End）

当"减少设置项"这个实现手段被当成"减少认知负荷"这个目的本身，AI 就失去了区分手段和目的的能力。

**Why:** 2026-04-08 WarmMagicSheet 讨论中，Claude Code 和两个 Gemini 实例基于 PRODUCT_SOUL 的"简单 > 功能多"否决了用户的"首次引导+偏好记忆"方案。三个 AI 的"共识"实际上是同一个输入的三次回声——Claude 读 PRODUCT_SOUL → 摘要发给 Gemini → Gemini 在同一框架内推理 → Claude 验证结论一致 → 伪共识。用户（唯一的异质信息源）用常识打破了这个回声室。

**How to apply — 三条操作规则：**

1. **手段-目的分离检查**：当基于产品原则否决方案时，必须回答：「这个原则关注的是结果（用户体验）还是机制（App 形式）？一次性配置消灭 100 次重复选择 = 做减法 = 符合"简单"原则。」

2. **类比还原为因果**：当讨论中出现美学类比（"徕卡不会这样做"），强制还原为因果链：「X 这样做 → 因为 ___ → 所以我们也应该」。如果"因为"填不出来或在本产品语境下不成立，类比应被抛弃。

3. **打破输入同质化**：向 Gemini 发送 prompt 时，不能只发"与当前方案一致的证据"。当用户提出与 AI 推理不同的方案时，必须在 prompt 中**平等呈现用户方案和 AI 方案**，而不是让 Gemini 评审 AI 预设的结论。
