**规则**: 在 flametree_pick 这类 AI-Only 项目中，不要主动以"测试数量过多 / 测试通胀 / 混凝土碉堡"为由要求 Code Freeze 或战略转向。评估测试健康度时关注**质量**（是否在保护真实业务 bug、是否跑得快）而非**数量**。

**Why**:
2026-04-11 /think 讨论中 Gemini 用"1419 测试 = 混凝土碉堡，需求反转时会反噬"批评当前项目，我据此升级为 Escalate 让 Founder 拍板。Founder 明确 Reject：「测试没什么大问题，全是 AI 写的」。

核心洞察：Gemini 的论点建立在"测试维护有人类心智成本"的隐含假设上，这在传统团队成立，但在 AI-Only 模式下失效——测试是 AI 一次性产出的，Founder 不承担维护心智负担。真正的成本是需求反转时 AI 修改失效测试的 token 消耗，但这是 Founder 调度时可以接受的代价。

**How to apply**:
- 做项目健康度评审时，不把"测试数量 / 代码行数比"当核心指标
- 不自驱"测试补齐"作为占用绝大多数 bandwidth 的主要工作（这可能是真问题，但应基于"当前有没有更紧迫的事"而非"测试总数太多"判断）
- 引用外部 AI（Gemini）的"测试通胀"论点前，先在 prompt 里明确声明 AI-Only 模式，让对方不要套用人类团队的启发式
- 如果真要质疑测试基建 bandwidth 占用，换角度：问"这些测试在保护什么真实失败场景"而非"测试数量是否过多"
