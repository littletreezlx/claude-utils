# AI 协作中的 Anti-Means-as-End 防御机制

## 教训来源

2026-04-08，产品讨论中发现系统性问题：AI 把产品文档中的实现手段（"不加设置项""键盘优先""拒绝云服务"）当成目的本身执行，而非追问真正目标。

**典型案例**：flametree_pick 项目中，"简单 > 功能多"原则被 AI 用来否决"首次偏好选择"方案（理由是"不应加设置项"）。但一次性配置消灭 100 次未来重复选择，是"净减少认知负荷"。

**问题性质**：这是 AI-Only 协作模式下的**核心通信协议冲突（Impedance Mismatch）**：
- Founder 端：用极具张力、绝对化、感性的语言传递产品愿景，在人类语境下高效
- AI 端：作为无机体执行者，将所有陈述视为字面量和硬性边界约束

## 根因分析

根因链：
1. 文档模糊 → 产生多个合理解读（A：无设置；B：少决策）
2. **缺少假设竞争机制** → AI 选第一个合理的，顺势深挖，产生内部一致性
3. 一致性被误认为正确性
4. **缺失的关键环**：AI 没有被要求区分"结果论证"（用户感受变化）和"机制论证"（App 形式变化）

**Means-as-End 的五种常见模式**：
| 模式 | 示例 | 问题 |
|------|------|------|
| 功能列举式 | "不做社交分享、排行榜" | 把手段当边界，而非防行为 |
| 美学类比式 | "像徕卡""不模仿大厂" | 因果链断裂，无中生有 |
| 绝对化措辞式 | "禁止纯黑""绝不" | 无例外条件，可被极端化 |
| 定量断言式 | "秒级响应" | 未经测量，误导优先级 |
| 情感启发式 | "宁可慢一拍" | 无数值边界，可无限放大 |

## 修正清单

### 修正 1：flametree_pick PRODUCT_SOUL.md Anti-Pattern 边界条件
- 一次性配置要满足：可逆性 + 必要性 + 可理解性
- 这是事后检查清单，不是推理过程中的动态权重

### 修正 2：flametree_music PRODUCT_SOUL.md Intent-Boundary Decoupling
采用"意图与边界解耦"策略：**不修改原句**，由 AI 追加 Engineering Boundary：

| 原句 | Boundary 追加 |
|------|--------------|
| 不破坏私密空间 | 针对"用户不知情的数据外传"，功能本身（如分享链接生成）不违反 |
| 不牺牲温度换效率 | "慢一拍"上限 400ms，仅核心交互适用，列表滚动不适用 |
| 不丢失设计个性 | 禁止无脑复制竞品交互模式；学习结果层并还原为因果链后落地是允许的 |
| Privacy-First | 改为"用户数据不主动上传到 Navidrome 以外服务端"；DNS/路由等基础设施经过第三方不在此限 |
| Local-First | 改为"操作写入本地后立即生效，无需等待网络往返" |
| 私密但开放 | 补充因果链：国际化 → 不依赖本地化文化特征 → 不泄露用户背景信息 |

### 修正 3：flametree_music ADR-003 Engineering Boundary 补齐
- "禁止 InkWell 涟漪" → "禁止默认 InkWell 涟漪，确有需求时用 NoSplash 工厂降级"
- "禁止纯黑" → "浅色模式下禁止纯黑，深色模式无此限制"

### 修正 4：docs/ui/theme.md 范围标注
- "所有阴影带棕色调，禁止纯黑" → "浅色模式下适用；深色模式阴影本就极轻，主要靠亮度而非色相分离层次"

### 修正 5：feat-discuss-local-gemini SKILL.md Anti-Means-as-End 条款扩展
触发条件扩展为 4 类：
1. 方案包含"一次性配置"或"用户偏好设置"要素
2. 出现品牌/产品类比（"像徕卡""像苹果"）
3. 原则中使用"禁止""绝不""必须""拒绝"等绝对化措辞
4. 出现功能列举式原则（如"不做 X 功能"而非"不支持 Y 行为"）

强制追问：「这个结论是基于用户体验的结果，还是基于 App 的实现形式？」
机会成本估算：要求估算 false negative 成本（「如果否决错误，损失是什么？」）

## 更通用的名字

**Rule-as-Sufficiency Fallacy**（规则当作充分条件谬误）
**Consequential Misattribution**（结果归因错误）
**Means-as-End Bias**（实现即目的偏见）

"纯度偏见"是 flametree_pick 教训时的项目内部名字，"Means-as-End"是更精确的通用术语。

## 防御层级总结

| 层级 | 修正 | 防御什么 |
|------|------|---------|
| 文档层 | Intent-Boundary Decoupling | 原则被过度泛化（保留原味，AI 追加边界） |
| Prompt 层 | 强制追问"结果 vs 机制" | 推理路径错误 |
| Skill 层 | 4 类触发条件 + 机会成本估算 | 否决成本为零的偏差 + 类比替代因果 |
