引用任何文档条目/产品原则去否决一个方案时，必须先区分「目的（用户体验结果）」和「手段（实现机制）」，然后用目的本身做裁决，而不是用手段做裁决。

**Why:** 在 flametree_ai 讨论「通用语音角色智能体引擎 合入 vs 独立」时，我引用 `docs/PRODUCT_SOUL.md` 的「键盘操作为核心」直接否决了合入方案，用户指出这是**手段**不是**目的**——真正的目的是「心流保护 + 减少切换摩擦」，语音同样能服务该目的（免手时刻、连续对话、移动场景）。flametree_ai 的铁律第 7 条明文警告了 Anti-Means-as-End 陷阱，但我仍然踩中了，因为**文档本身把手段包装成了核心定位**（Line 12: "键盘操作为核心"），AI 直接把文字当约束执行，没有还原因果链。

**How to apply:**
1. 当下次基于任何文档条目/产品原则否决方案时，完成三步检查：
   - 还原因果链：「违反 X → 通过什么机制 → 导致什么用户体验变差」
   - 区分层级：如果 "违反 X" 的内容是实现机制（"不加设置项"、"键盘优先"、"单主题"、"某种布局风格"），停下并追问「这条机制服务什么目的？」
   - 用目的本身裁决：检查新方案是否让那个目的真的变差。如果因果链断裂或方案同样服务该目的，**不能**拿这条文档做否决理由
2. 发现文档本身把手段写成目的时，**主动提议修改文档**，把手段性表述改为目的性表述。不修正源头的话，未来会继续污染判断
3. 类比（"徕卡不会这样做"、"VS Code 就是这样的"）不是论证，必须还原为「X → 因为 ___ → 所以我们也应该」的因果链，因果断裂则抛弃类比
4. 这条规则在 AI-Only 协作模式下尤其关键——文档就是可执行的约束本身，AI 不会像人类那样灵活变通。因此措辞精度决定 AI 决策质量
