进入任何技术方案讨论前，必须先确认并写明运行环境的硬事实：主交付平台（iOS / Android / macOS / Windows / Web）、使用模式（前台常驻 / 后台运行 / 锁屏唤起）、设备形态（手机 / 平板 / 车机 / 桌面）、网络环境（常连 / 断续 / 离线）。不要在这些事实未验证前就开始堆砌技术方案。

**Why:** 在 flametree_ai 语音模态讨论中，我连续好几轮都基于"iPhone 主交付 + 可能被系统杀后台 + 需要 CallKit 集成"的假设展开深度技术讨论，甚至让 Gemini 也围着 iOS 后台保活打转。最后用户告诉我"ios 保活不是重点！实际我都是开在前台的，而且我实际用的 android 机器！说 ios 只是方便你在我的开发机 mac 上测试"——整个 iOS 后台保活 / CallKit 集成的讨论都是在错误的前提上做的，浪费了大量 token 和用户耐心。**根源错误**：我默认"用户在 Mac 开发 → 主交付是 Apple 生态"，但开发机 ≠ 生产设备。用户可能在 Mac 上写代码，但主力设备是 Android 手机，主场景是前台常驻，根本不关心 iOS 后台。

**How to apply:**
1. 讨论任何涉及"客户端/用户真实使用"的技术方案前，第一件事必须问：**"这个功能实际在什么设备上跑？什么操作系统？用户用的时候是前台还是后台？屏幕开着还是息屏？"**。不要等到方案已经写了 3 轮才发现主平台错了
2. **开发机 ≠ 生产设备** 是铁律：用户在 Mac / Windows / Linux 上开发不等于主交付平台是同一个；Flutter 项目尤其容易掉进这个陷阱，因为 Flutter 跨端，开发时默认跑 macOS 最方便
3. 当产品讨论里出现"开车"、"跑步"、"做饭"、"上下班通勤"这类真实物理场景时，必须立刻切换心智：**用户不在键盘前，不在开发机前，很可能用的不是 Apple 设备**
4. 如果用户之前隐约提过主平台但我记错了，回到对话历史 grep 检查，不要用"最近讨论到的平台"代替"真实主平台"
5. 遇到"前台/后台""常驻/唤起""屏幕开/关"这类状态假设时，必须明确标注是"我在假设"还是"用户确认过"
