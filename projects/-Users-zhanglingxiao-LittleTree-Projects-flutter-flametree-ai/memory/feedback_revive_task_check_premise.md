复活已搁置任务、继承旧规划、激活 backlog 条目之前，**编码动手前**必须先核验该任务的设计前提在当前代码库是否仍成立。具体做法：grep 或 Read 任务描述里提及的所有"特殊配置 / 关键能力"，判断现有代码是否已具备；若已具备，该任务可能已冗余或需要大幅重写。

**Why**: 2026-04-18 D17 案例。Claude Code 在 D16 /think 讨论中仓促把"立即启动 P0B-1~P0B-6"作为结论的一部分推上去，照搬了 Phase 0A 时期写的任务描述。其中 P0B-2（Flutter dev-only 测试页改造）规划的所有"特殊配置"——`VOICE_COMMUNICATION` 音源、`wakelock_plus`、断线重连、全屏电话界面——现有 Phase 1 APK 代码（`app/lib/features/voice_agent/`）已全部具备。Founder 的一句"我已经在手机 APP 上测试过了，你这个相比我的安卓 APP 的环境有什么区别？"直接戳穿了前提过时。若不是 Founder 质疑，本可能花费 1-2 天时间建一个注定冗余的测试页。

**How to apply**:
- 触发时机：看到任务标签"补课 / 继承自 Phase X / 旧规划重新激活 / backlog 挖出"时高度警觉
- 第一步：把任务描述里提及的每个"特殊能力 / 专用配置"列出来
- 第二步：用 Grep / Read 核查这些能力在当前代码库是否已实现；注意对比的是**当前 HEAD**，不是写规划时的 HEAD
- 第三步：若 ≥50% 的规划配置已具备，必须停下来复核该任务整体是否仍必要，而不是"只补缺失的那部分"——往往整个任务的设计前提已经失效
- 反模式：看到"这是 D16 定的结论"就直接开工；决策文件里的"立即启动"不豁免前提核验
- 与"think-gem-project"联动：若 /think 输出中出现"立即启动 <旧规划任务>"，在采纳前单独做一次"前提仍成立吗"扫描
