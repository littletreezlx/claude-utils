"不做"清单应该只包含**绝对负面**边界（永远不做 / 本项目不碰 / 产品层决策否决）。凡是"推迟到 Phase 2"或"未来阶段会做"的能力，**不要**列进"不做"清单——应该直接写进对应 Phase 的任务列表。同时，当 Claude 或外部 AI 的建议是"推迟到 Phase 2"（而非"完全砍掉"）、且 Phase 2 规划已经存在、且没有产品层冲突时，**直接写进计划**，不要反复问用户"你确认砍掉吗？"。

**Why:** 在 flametree_ai TODO.md 的"决策锁定边界"里，我写了 "❌ 不做 Wake Word 常驻监听（推迟到 Phase 2）"、"❌ 不做 Barge-in（推迟到 Phase 2）" 这类条目。用户直接点破："为什么不做 Barge-in（推迟到 Phase 2），如果确定将来要做我觉得你可以不用这么回答，直接写到计划就行。" 这种表述既矛盾（不做 vs 推迟到 P2），又制造了虚假的决策点（用户不得不反复点头说"确认推迟"），浪费用户的决策带宽。反复确认本质是 AI 推卸决策责任。

**How to apply:**
1. "不做"清单的入选标准：**永久不做** 或 **本次项目明确不碰** 或 **产品层定位冲突必须否决**。凡是"现在不做，将来会做"的能力一律不进此清单
2. 如果某能力被推迟到 Phase N（N > 当前 Phase），直接写进 Phase N 的任务列表即可。任务列表本身就是规划，不需要再在"不做"里重复
3. 当 Gemini / Claude 的建议是"推迟"而非"砍掉"，且三个条件满足（未来 Phase 规划已存在 + 推迟理由充分 + 无产品层冲突），**直接落进计划**，只在响应里简短说一句"我已经把 X 移到 Phase N"即可，不要反复向用户求确认
4. 需要用户拍板的只有两类：**完全砍掉某个核心能力** 或 **产品层定位反转**。"推迟到 Phase 2" 既不砍又无反转，属于可以直接决策的范畴
5. 当"不做"清单里出现矛盾表述（不做 X + 推迟 X 到 Phase 2 同时存在），立即清理。清理方式：从"不做"移除，确认 Phase 2 列表里有，没有就补上
