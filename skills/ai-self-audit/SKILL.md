# AI Self-Audit — AI 自主维护能力审计

## 目的

审视项目对 AI 的"人机工效学"（AI-Ergonomics）：知识是否显式、边界是否清晰、反馈是否闭环。一个全新 AI 会话能否在最少人工干预下理解、修改、测试、交付这个项目？

**前提**：本体系是 AI-Only Development——AI 全权负责代码，人类是产品负责人。不存在"人类开发者兜底"的安全网，AI 的知识盲区就是项目的盲区。

**与其他工具的边界**：consistency-check 查"现在对不对"，health-check 查"项目健不健康"，**ai-self-audit 查"上下文对 AI 是否友好"**。

## 触发条件

用户明确请求时触发（重型操作，不自动触发）：
- "如何让你更好维护这个项目" / "项目自省" / "self-audit"
- "怎么让 AI 更好的独立工作" / "AI 维护性"
- token 充足时用户主动要求深度思考项目可维护性

## 核心方法论：逆向溯源（非"假装新手"）

**铁律**：AI 无法真正"假装自己是新手"——已加载的上下文会导致虚假繁荣（"上下文清晰，无问题"）。

正确的审计方式是**逆向溯源**：
> "我理解了 X。但 X 的解释明确写在哪个文件里？如果答案是'我从代码推导出来的'或'我从对话历史中知道的'，那就是隐式知识——新 AI 会在这里抓瞎。"

每个审计发现必须标注知识来源：
- **显式** — 写在 CLAUDE.md / docs / 代码注释中，新 AI 可直接获取
- **隐式** — 需要读多个文件推导、依赖领域常识、或依赖对话历史，新 AI 大概率会踩坑

## 四维审计框架

### 1. 上下文入口与可导航性

CLAUDE.md 到真实代码的"指针"是否完整且未破损？

**逆向溯源检查**：
- 列出理解项目所需的核心前置知识（技术栈、架构模式、构建命令、关键约定）
- 逐一验证：每条知识是否在 CLAUDE.md 或 docs/ 中有明确出处？
- FEATURE_CODE_MAP 中的路径是否全部存在？（Glob 验证）
- 关键入口点（路由、事件处理器、main）能否从文档直接定位？

**就地修复**：发现破损指针或缺失条目，直接更新 CLAUDE.md / FEATURE_CODE_MAP。

### 2. 隐式知识显性化程度

项目中有多少"只在 Founder 脑子里"的知识？

**逆向溯源检查**：
- 扫描代码中的 HACK/FIXME/WORKAROUND/XXX 注释——有上下文说明 why 吗？
- 扫描 git log 中的 revert 和连续 fix 模式——暗示 AI 反复踩同一个坑？
- 反直觉的设计决策（如不用常规方案而用特殊方案）是否文档化了原因？
- 领域术语和业务规则是否有记录？（专有名词不解释 = 新 AI 猜错概率极高）
- docs/ 中是否有假设人类开发者的表述？（"开发者需要"、"手动运行"、"请确认后继续"等会误导新 AI 会话跳过本该自主完成的步骤）

**就地修复**：发现缺失的 why，直接在代码注释或 CLAUDE.md 中补充。发现未记录的术语，直接追加到 CLAUDE.md。

### 3. 验证反馈环

AI 改完代码后，系统能否"立即打脸"——告诉它改对了还是改错了？

**幽灵质量关卡扫描**（AI-Only 项目特有）：
扫描 CLAUDE.md、skills、docs 中是否残留"人工审核"、"人工走查"、"等待 review"等步骤——这些在 AI-Only 项目中永远不会执行，是验证链路中的断裂点。发现即直接修复为 AI 可执行的替代方案。

**三道防线对照**（全局 CLAUDE.md 定义的理想状态）：
1. 结构性时间差 — 测试和代码是否有认知隔离？
2. 运行时验证 — 有无 Debug State Server 或等效的运行时探针？
3. 红队对抗 — 核心模块是否接入 `/test-verify`？

**检查思路**（因项目类型而异）：
- 测试/lint/构建是否一键执行、无手动步骤？
- 验证链路中有没有"只能靠人看"的断点？
- 测试数据能否一键重置到已知基线？（脏数据是自闭环的头号杀手）

| 项目类型 | 验证手段示例 |
|---------|------------|
| 有 UI 的 App | Debug State Server + curl 验证状态 + 截图/语义树 |
| Web 应用 | Playwright/浏览器自动化 + API 端点验证 |
| CLI/脚本 | 直接执行 + 比对 stdout/文件输出 |
| 库/SDK | 单元测试 + 集成测试已足够 |
| 后端服务 | curl API + 检查 DB 状态 + 日志断言 |

**产出**：验证链路中的断点写入 TODO.md（需要代码实现）。

### 4. 上下文碎片化程度

修改一个逻辑需要同时加载多少个文件？这是 LLM 独有的痛点——过度拆分对人类是文件夹管理，对 AI 是 Token 爆炸和注意力分散。

**检查思路**：
- 核心功能的修改半径：改一个功能通常需要动几个文件？散布在几个目录？
- 是否存在过度抽象：简单逻辑被拆成多个接口层、adapter、wrapper？
- 高频修改区的文件是否就近放置？

**产出**：碎片化严重的模块写入 to-discuss.md（需要用户决策是否重构）。AI 不主动建议大规模重构——只标记问题和理解成本。

## 执行流程

### Step 1: 逆向溯源扫描
对四个维度逐一检查。每个发现必须包含：
- **具体证据**（文件路径、代码片段、命令输出）
- **知识来源标注**（显式 vs 隐式）
- 不主观臆断——没有证据的不算发现

### Step 2: 瓶颈分级
- **🔴 阻塞性**：新 AI 会话完全无法独立完成某类任务
- **🟡 摩擦性**：能做但效率低或易出错
- **🟢 优化性**：已能工作，可以更好

### Step 3: 就地修复 + 分流

**直接执行**（本次会话完成）：
- 补充缺失的 CLAUDE.md 内容、修复破损指针
- 给缺少 why 的 HACK 注释补充上下文
- 追加未记录的领域术语/业务规则

**写入 TODO.md**（需要写代码）：
- 验证链路中的断点（如缺少 seed 脚本、Debug Server）
- 缺失的关键测试

**写入 to-discuss.md**（需要用户决策）：
- 上下文碎片化严重的模块（是否值得重构）
- 架构级改进建议

### Step 4: 输出审计报告

```
## AI 自主维护能力审计

**综合评级**: 🟢/🟡/🔴

### 各维度评估
| 维度 | 评级 | 关键发现 |
|------|------|---------|
| 上下文入口 | 🟢/🟡/🔴 | ... |
| 隐式知识 | ... | ... |
| 验证反馈环 | ... | ... |
| 上下文碎片化 | ... | ... |

### 隐式知识清单（最高价值产出）
| 知识点 | 来源 | 已修复？ |
|--------|------|---------|
| "用 X 而非 Y 是因为 Z" | 隐式（代码推导） | ✅ 已补充到 CLAUDE.md |
| ... | ... | ... |

### 已执行改进
- [改进 1]（文件路径）

### TODO.md / to-discuss.md 新增
- [条目]
```

## 踩坑点

- **知识的诅咒是头号敌人**：永远不要"假装新手"，用逆向溯源——"我懂的这些，出处在哪？"
- **不要变成 consistency-check**：不查文档和代码是否一致，查的是"知识对 AI 是否可获取"。
- **不要评价代码优雅度**：只关注理解成本，不评判设计质量。屎山代码如果注释清晰、AI 能改，就是合格的。
- **克制重构冲动**：碎片化问题只标记不修，交给用户决策。AI 最容易在这里过度建议。

## 约束

- 不修改业务代码逻辑（可以加注释、更新文档）
- 文档/注释类改进直接执行，代码结构问题只建议
- 报告务实：不夸大问题，不虚报"已优化"
- 遵循三文件分流：事实 → TODO，观点 → to-discuss，材料 → _scratch
