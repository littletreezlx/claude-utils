# Quarterly Review — 个人季度/年终述职端到端流程

## 用途

把"个人季度述职"从一次性写作变成一套可在 Q2/Q3/Q4 / 年终复用的工作流，确保：
- **基调与真实工作分布对齐**（飞书事实先行，禁止 AI 预设主线）
- **承接连续叙事**（每季开篇承接上季/年终立下的方向）
- **风格客观**（禁止把对话指令写进文档、禁止过度修辞）
- **诚实差距**（承认差距是加分项，掩饰是减分项）

## 触发条件

满足任一即触发：
1. 用户明确说"准备季度述职 / 做季度述职 / Q2 述职 / 年终述职 / 季度复盘"
2. 在述职项目目录（含 `XXXX述职.md` 或 `XXXX年终总结.md` 等历史文件）中开启新季度工作
3. 用户提到"老板要看述职"、"总监要求复盘"、"季度产出复盘"

## 执行流程（8 步骨架）

| 步 | 动作 | 产出 |
|---|---|---|
| 1 | **项目上下文初始化**：改造 `CLAUDE.md`，加入本季硬约束（总监指令、可能受众、季度 vs 年终差异） | 更新后的 CLAUDE.md |
| 2 | **目录整理**：建 `_参考/`（历史下沉，只读）+ `<本季>/` 工作区（飞书拉取/工单复盘/技术文档/述职定稿子目录） | 干净的目录 |
| 3 | **基调对齐 think**（推荐但非必须）：调 `/think` 验证规划是否过早收敛叙事 | think Digest + 决策记录 |
| 4 | **飞书拉取**：从 feishu MCP 拉本季工作项（需求/缺陷/复盘/工单）+ 个人创建文档，分类保存到 `<本季>/飞书拉取/` | 4-5 份分类素材 md |
| 5 | **三因子分析与大纲收敛**：对素材做横切聚类，应用『投入占比 × 结果强度 × 管理关注度』决定主线，输出 `00-素材综合汇总.md` | 大纲推荐 + 信息缺口清单 |
| 6 | **Founder 拍板 + 补齐信息缺口**：确认大纲结构、补客户专项 / 灰度策略 / Q2 方向等关键事实 | `撰写前决策.md`（撰写阶段的硬约束） |
| 7 | **撰写 + 老板提问预演**：按决策书撰写 → 逐章自检『好的标准 / 方向对不对 / 差距多少』→ 不够透明的地方补 baseline 注脚 | `<本季>季度述职.md` |
| 8 | **飞书云文档化**：用 feishu MCP 推到飞书 | 飞书云文档链接 |

详细流程清单见 [`references/workflow-checklist.md`](references/workflow-checklist.md)

## 核心方法论（不可妥协）

1. **事实先行**：飞书数据未盘点前，禁止预设主线 / 章节位置 / 篇幅占比。只能预设『必须出现什么』的硬约束（如总监指令）
2. **主线三因子规则**：主线 = 投入占比 × 结果强度 × 管理关注度。三因子任一为零则不入主线；管理关注 ≠ 主线
3. **老板视角是验收锚而非写作模板**：老板三问『好的标准 / 方向 / 差距』集中在差距章 + 定稿前预演，禁止全文逐段注脚化
4. **承接连续叙事**：每季开篇承接上次年终/上季度立下的方向，避免述职变成孤立切片
5. **基线类成果反向 framing**：不要把『建立 X 基线』包装为成果，要 framing 为『X 让哪些治理可持续』，否则会暴露过去无度量的窘迫
6. **客户切换支撑可独立成章**：如果某个大客户切换是本季工作主线之一（如本次冰雪时光），单独成章而非散落在各处

详细 framing 原则与正反例措辞见 [`references/section-framing.md`](references/section-framing.md)

## Gotchas / 踩坑点（每用一次更新）

| 坑 | 失败模式 | 修正 |
|---|---|---|
| 把对话指令写进文档 | 出现『响应总监讲变化』/『诚实说明』等把 AI 思考暴露在正文的措辞 | 写完通读，删掉所有"对老板/总监说话"的元描述 |
| 表格里 `<br/>` | 飞书云文档不支持，本地预览也丑 | 表格里禁止 `<br/>`，要么拆成多个段落要么合并单段 |
| "wiki" 当口语 | 文档里出现『10 篇 wiki』 | 一律改为"文档"；链接文本用文档名 |
| 把开发期 bug 叙事为"闭环 N 项关联缺陷" | 听起来像线上事故修复，实际只是测试期发现的常规 bug | 平实写：开发过程发现并修复 X 项 bug，不要"闭环"措辞 |
| 上季度文档重复列 | 离线收银 6 篇方案文档去年贴过本季又贴 | 文档建设章只列**本季新增**的（与上季未重叠的） |
| PO / 全面负责类 framing 过度 | 『本人全面负责 X』、『作为 PO 主导...』听起来甩锅或邀功 | 改为客观的『在 X 线上承担了 PO 角色』、『参与了部分产品设计』 |
| 还在开发中的方案被当成果展示 | 把"打印模块可靠性保障机制"这种半成品方案当核心设计强调 | 在开发中的方案要么淡化提及、要么留到下季度再讲 |
| 灰度发布等结构性短板被遗漏 | 只看缺陷数据不看发版策略，错过『发版无灰度 / 0 订单使用』这种真实差距 | 差距分析必问：本季是否有"虽然没出事但暴露了机制缺失"的事项 |
| 创建飞书文档后未回写 doc_id | 后续要修改/重发布时找不到原文档，只能新建一份，造成飞书侧多份重复 | 飞书 `create-doc` 成功后**立即**把 `doc_url` + `doc_id` 写到独立的 `PUBLISH.md`（同目录），**禁止污染述职正文文件**——元信息是基础设施，不应混进给老板看的内容 |
| 故障等级凭标题脑补 | 看『退登仍出票（爆单）』标题就脑补成 P0/P1，看『效期标签打不出来』就脑补成『打印模块历史债务』；事后被 Founder 逐条拆穿（实际是 S3 + WebView 处理 H5 的历史债务） | 写质量复盘前**必须**用 feishu MCP 拉 case_study 详情拿真实 severity + 真实根因。禁止凭标题词推断。等级以飞书 case_study 字段为准 |
| 完成度数据未核实就编 X/Y | 编出『6 大子模块完成 1/6，其余在途』作为完成度，实际 Founder 说『都上了』，框架性误判 | 完成度/进度数据撰写前必须问 Founder 真实状态；禁止编 X/Y 假精度 |
| 个人成长 = 固有能力 | 把『PO 角色补位 / 从单纯实现转向方案制定』当 Q1 成长，但 Founder 长期就在做 PO 和方案主导，等于把固有能力包装成新成长 | 个人成长每条都自问：『这是 Q1 才有的，还是我一直在做的？』固有能力 ≠ 季度成长，是技能背景。详见 [section-framing.md § 7.5](references/section-framing.md) 三层判定 |
| 部门基线当个人贡献 | 把『AI 编码提效已成日常』作为 Q1 价值，但部门人人都在用，零差异化 | 写贡献前自问：『这是我独有的，还是部门所有人都在做的？』后者是部门基线，不算 personal contribution |
| 戏剧化数字对比反伤 | 『100 次发版 vs 0 订单』作务实陈述写出，反成『100 次干嘛了？』的负面联想钩子 | 描述未启用/未达成状态时，不要用张力数字组合（X 次 vs 0 类型），改用『节奏在推进中』『等待业务侧切换』这类去戏剧化措辞 |
| 提议的兜底方案前提未验证 | 提议『内部门店软验证作为灰度兜底』作为差距改进举措，但 Founder 公司根本没有"内部门店"概念 | 提议改进举措前先问 Founder：『前提条件 X 在你们这边成立吗？』禁止用 AI 训练数据里通用的兜底套路硬塞 |

## 约束

- **事实在飞书，不要凭记忆**：每季实际工作分布只能靠飞书数据盘点出来
- **下游 AI 容易被预设牵着走**：CLAUDE.md 中只放流程骨架与方法论，**禁止预设叙事性结论**
- **不自动推飞书**：定稿后由 Founder 确认才能 publish
