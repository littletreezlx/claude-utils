# 流程检查清单（每季对照走一遍）

## Step 1：项目上下文初始化

- [ ] 把项目 `CLAUDE.md` 的标题、时间窗、汇报范围切换到本季
- [ ] 加入本季总监指令（原话引用 + 落地解读）
- [ ] 加入本季可能的特殊受众（如老板可能会看 → 老板视角验收）
- [ ] 保留稳定流程骨架，**禁止写入叙事性结论**（如"X 必须独立成章"、"Y 必须置顶"）
- [ ] 已沉淀的方法论保留（事实先行 / 三因子规则 / framing 原则等）

## Step 2：目录整理

- [ ] 创建 `_参考/`，把跨季复用的样板（述职要求、解读、上季定稿）移入
- [ ] 创建 `_参考/<上一年>年素材/`，把上一年的工单复盘 / 技术文档存档下沉
- [ ] 创建 `<本季>/`（如 `2026Q2/`）作为工作区，含 4 子目录：
  - `飞书拉取/`
  - `工单复盘/`
  - `技术文档/`
  - 终稿 `<本季>季度述职.md`

## Step 3：基调对齐 think（推荐）

- [ ] 调 `/think` 验证当前规划是否过早收敛叙事（特别是有总监指令时）
- [ ] 把 think Digest 与决策结论沉淀到 `<本季>/撰写前决策.md`（可选）

## Step 4：飞书拉取

- [ ] `search_project_info` 确认空间名（shipd / shihengconsole）
- [ ] `search_user_info` 拿 user_key（用于 MQL）
- [ ] `feishu-docs/get_user` 拿 open_id ou_xxx（用于文档搜索）
- [ ] `list_workitem_types` 看可用类型
- [ ] 拉 4 类工作项（需求 / 缺陷 / 复盘 / 工单），每类保存独立 md
- [ ] 拉本季个人创建的飞书文档，剔除草稿/副本/个人参考文档

## Step 5：素材综合汇总与三因子分析

- [ ] 横切聚类（性能 / 客户专项 / 业务主线 / 其他）
- [ ] 加权估算投入占比（P0=4, P1=3, P2=2 加权）
- [ ] 评估每个候选主题的『结果强度』（量化数据 / 客诉降幅 / 文档沉淀）
- [ ] 标注每个主题的『管理关注度』（总监 / 老板显性关心）
- [ ] 应用三因子规则收敛主线，推荐三种典型结构之一
- [ ] 列出**信息缺口**，标注哪些需要 Founder 补充

## Step 6：Founder 拍板 + 补齐信息缺口

- [ ] 大纲结构由 Founder 确认（结构 1 / 2 / 3）
- [ ] 补齐高重要性缺口（特别是承接连续叙事相关：上季 flag 的本季进展）
- [ ] 关键事故的定性（线上事故？历史代码？）
- [ ] Q2 / 下季方向（用 Founder 原话）
- [ ] 输出 `<本季>/撰写前决策.md` 作为撰写阶段硬约束

## Step 7：撰写正文

### 7.0 事实核验前置（撰写前的硬门槛，违反 = 返工）

- [ ] **故障 severity**：凡涉及『线上事故 / 故障级别（P0/P1/S2/S3 等）』，必须从飞书 case_study 字段拉取真实 severity，禁止凭标题词（如『爆单』『重复出票』）脑补
- [ ] **case 根因**：每条线上 case 的根因描述，必须读 case_study 详情页内容，禁止凭标题推断（标题『效期标签打不出来』≠ 打印模块债务，可能是 WebView/H5 历史债务）
- [ ] **完成度 X/Y**：写『N 大子模块完成 X 项』前，必须问 Founder 真实状态。禁止编 X/Y 假精度
- [ ] **个人成长 vs 固有能力**：每条『个人成长』bullet 前自问：『这是 Q1 才有的，还是我一直在做的？』固有能力删
- [ ] **个人贡献 vs 部门基线**：每条『个人贡献』bullet 前自问：『这是我独有的，还是部门人人都做的？』部门基线删

### 7.1 章节撰写

- [ ] 开篇承接上季方向（含进展状态 + 未推进项的诚实说明）
- [ ] 工作成果按主题切片，篇幅按真实投入分配
- [ ] 客户切换支撑独立成章（如适用）
- [ ] 性能 / 基线类按 framing 原则（治理升级支撑而非主成果）
- [ ] 差距分析按"现状 → 根因 → 改进"三段式
- [ ] 改进举措的前提条件必须在 Founder 的真实环境下成立（不要塞通用兜底套路如『内部门店软验证』，先确认有没有这个概念）
- [ ] 质量复盘含 baseline 对照（vs 上季 / vs 上年）
- [ ] 文档建设只列本季新增（带链接）
- [ ] Q2 方向具体到可执行动作（用 Founder 原话）

### 7.2 措辞审查

- [ ] 不用戏剧化数字对比（X 次 vs 0 类型）描述未启用状态——会变成负面联想钩子，改用『节奏在推进中』这类去戏剧化措辞
- [ ] 不在正文 mix 英文（如『能力 ready，不是业务价值 ready』），改纯中文表述

## Step 8：老板提问预演自检

逐章自问『好的标准是什么？方向对不对？差距多少？』，能答上来就过。答不上来：
- 在该章节加 1-2 句基准 / baseline 注脚（不展开）
- 或在差距章集中回应

## Step 9：Founder 审稿（不可省）

- [ ] 通读全文，检查是否出现以下踩坑（详见 SKILL.md Gotchas 表）：
  - 把对话指令写进文档
  - 表格 `<br/>`
  - "wiki" 当口语
  - bug 被叙事为"闭环"
  - 上季文档重复列
  - PO / 全面负责类 framing 过度
  - 还在开发中的方案被当成果
  - **故障 severity / 根因凭标题脑补未拉飞书数据**
  - **完成度数据 X/Y 未核实就编**（如『6 大子模块完成 1 项』）
  - **个人成长项实际是固有能力**（Founder 长期就在做的事）
  - **部门基线（如 AI 编码提效）当个人贡献**
  - **戏剧化数字对比（X 次 vs 0 类型）造成负面联想**
  - **改进举措的前提条件在 Founder 真实环境不成立**（如『内部门店软验证』但其实没这个概念）
  - **正文 mix 英文**（如『能力 ready』），改纯中文表述
- [ ] 数据一致性扫描：跨 5+ 章节核对核心计数（故障数 / 故障等级 / 完成度 / 工单数）是否前后一致——一处变更易导致多处过时
- [ ] Founder 反馈后做必要重写（很可能要大改 1-2 章）

## Step 10：飞书云文档化

- [ ] Founder 确认定稿后，用 feishu MCP 创建飞书云文档
- [ ] 检查飞书表格渲染（特别是 `<br/>` 是否被去除）
- [ ] 把链接附给 Founder
