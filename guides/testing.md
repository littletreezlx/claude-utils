## 精准测试命令

```bash
# ✅ 指定文件 + 静默模式
jest path/to/specific.test.ts --silent --no-coverage
pytest path/to/test.py -q --tb=short
../scripts/test.sh test/specific_test.dart   # Flutter 项目用统一脚本

# ❌ 无脑全量测试
jest / pytest / flutter test
```

## 测试优先级

- **必须**：核心业务流程、复杂业务逻辑、状态机转换
- **可选**：工具函数、简单计算、数据转换
- **一般不需要**：getter/setter、纯展示组件、配置常量

## 测试驱动重构信号

写测试时遇到以下情况 → 代码需要优化：

- 难以 mock → 依赖太多，需要依赖注入
- setup 复杂 → 接口设计问题，职责不单一
- 无法隔离 → 模块耦合，需要拆分边界

> 难测的代码 = 难维护的代码

## AI 测试陷阱深度解析

### 陷阱 1：断言当前行为

```
// ❌ AI 看到代码返回 null，就写 expect(result).toBeNull()
// 但 null 可能就是 bug，正确行为应该返回 []
```

**防御**：写测试前先问"预期行为是什么"，不是"当前行为是什么"

### 陷阱 2：过度 Mock

```
// ❌ Mock 了整个数据库层，测试永远通过
// 真实环境 SQL 语法错误根本发现不了
```

**防御**：核心数据路径用集成测试，只 Mock 外部 API 和不稳定的 I/O

### 陷阱 3：断言太宽泛

```
// ❌ expect(result).toBeDefined()  // 返回 {} 也通过
// ❌ expect(list.length).toBeGreaterThan(0)  // 返回错误数据也通过
// ✅ expect(result).toEqual({ id: 1, name: 'test' })
```

## 测试与调试的交叉原则

### 分层诊断顺序

排查问题时，按数据流方向逐层验证：

1. **数据层**：数据是否正确接收？（Network 日志）
2. **状态层**：状态是否正确更新？（关键状态日志）
3. **UI 层**：渲染是否正确？
4. **动画层**：特效是否正确？（优先保证功能）

### AI 可观测性

- 优先"看日志"而非猜测：通过项目脚本抓取崩溃日志/错误堆栈
- Bug 修复闭环：读日志定位 → 修复 → 重新运行验证 → 对比前后输出

### 日志规范

- 只记录关键状态变更，避免循环中打印
- 跨组件调用必须有进出日志
- 异步操作必须有完成确认日志
