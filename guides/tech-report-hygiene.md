# 技术调研稿 / 对外稿撰写卫生

本指南面向 **调研报告、可行性研究、竞品对比稿、架构综述** 等"发给产品方向 / 运维团队 / 管理层"的技术文档。日常工程文档（CLAUDE.md、设计文档、代码注释）不强求适用。

核心原则：**读者不是你的同僚 AI**。对外稿服务于他人决策，不是服务于 AI 的记忆或作者的自我表达。

---

## 硬规则（同步于 `~/.claude/commands/techdoc.md`）

- 禁止代码、类名、文件路径、类数量
- 禁止元信息和交叉引用（"本文档面向…"、"实施细节见…"、"以下章节…"）
- 禁止代码视角措辞（「无 X 代码」「无 X 痕迹」「客户端可见度」「反编译确认」）
- 断言优先，不用防御性推测词（「极可能 / 大概率 / 完全可以 / 疑在」）
- 推荐必须基于已验证事实或明确声明的假设

硬规则违反就是交付缺陷，无回旋余地。

---

## 软规则（建议，有场景判断空间）

### 1. 自谦过头也不专业

对产品读者写「我们不是媒体直播公司,不具备大规模运维能力」反而让读者怀疑团队。技术约束要用事实陈述，不要团队自嘲：

- ❌ 「我们不是媒体直播公司,不具备大规模分布式媒体服务器的运维能力与预算」
- ✅ 「媒体服务器的规模化运维不在团队当前建设重点内」
- ✅ 「每路并发对服务器入向带宽、CPU、端口池是乘法关系的成本」

### 2. 证据稿与决策稿的分工

调研类文档往往有两种角色，混在一起会互相拖累：

| 文档类型 | 职责 | 放什么 | 不放什么 |
|---------|------|--------|---------|
| **证据稿 / 对比稿** | 呈现事实 | 架构图、技术要点、能力矩阵、触发流程对照 | 我方推荐、决策倾向 |
| **决策稿 / 可行性报告** | 推导决策 | 方案对比、推荐、待拍板问题、资源诉求 | 证据细节（外链到证据稿） |

实操：证据稿上传飞书独立 URL，决策稿内嵌 2-3 条摘要 + 外链 + 决策推论。**不要把证据稿的详细表格复制进决策稿**——复制粘贴的文档未来必然 drift。

### 3. 推荐前先反问自己

给任何"我们建议走 X 方案"的推荐前，过一遍自检：

- 这条推荐依赖哪些**前提**？前提是已验证的还是我假设的？
- 如果前提不成立，推荐是否立刻垮掉？
- 有没有工程量 / 成熟度 / 时间成本的量化对照？（不是纯感觉）
- 对比方案的理由是否对称？（不能只挑支持自己方案的论据）

真实案例（2026-04-17）：推荐时写了「食亨场景倾向 qm 模式：有自有运维 + 客服体系」，前提「有自有运维后台」是假设。用户一挑：「配对码模式（kry）是不是更成熟、更容易实现？」——推荐立刻垮掉。实际配对码模式工程量小 3-5 倍、对运维组织无假设，更适合 MVP。

教训：**别把"我希望成立的前提"当成"已经成立的前提"**。

### 4. 章节编号连续

对外稿不要留章节跳号（§3 → §5 会让读者误以为丢了一章）。删段后整体重编。

## 工程手法

### 删段后章节重编号脚本

```python
import re

path = 'path/to/doc.md'
THRESHOLD = 5  # 从被删章节的下一个章节号开始减 1

with open(path) as f:
    content = f.read()

def dec_ref(m):
    n = int(m.group(1))
    sub = m.group(2) or ''
    return f'§{n-1}{sub}' if n >= THRESHOLD else m.group(0)

def dec_heading(m):
    prefix, num, sub = m.group(1), int(m.group(2)), m.group(3) or ''
    return f'{prefix} {num-1}{sub}' if num >= THRESHOLD else m.group(0)

# 必须先改引用再改标题，否则标题里的数字会匹配 §N（因为没有 § 前缀，其实不冲突，但顺序保守）
content = re.sub(r'§(\d+)(\.\d+)?', dec_ref, content)
content = re.sub(r'^(##|###) (\d+)(\.\d+)?', dec_heading, content, flags=re.M)

with open(path, 'w') as f:
    f.write(content)
```

执行后 grep 验证：`^## \d+\.` 是否连续、`§\d+` 引用是否都更新。

### 上飞书前的 lint checklist

推送前 grep 扫一遍，有匹配就修：

```bash
grep -nE '反编译|APK 反编译|客户端代码|无 .* 代码|无 .* 痕迹|客户端可见度|极可能|大概率|完全可以|疑在|我们不是|本文档面向|以下章节|以下事项|\.so|com\.[a-z]+\.' doc.md
```

扫出来的每一条都要判断：是事实陈述还是暴露视角？不确定就换成归属性肯定措辞。

### 上飞书前的标题与正文 H1 处理

飞书通过 `title` 参数自动生成文档一级标题，正文中的 `# 标题` 会被视为重复。推送时：

- 从首行 `# 标题` 提取 title 参数
- 正文 markdown 去掉首行 H1 后传入（`tail -n +3 doc.md`）

---

## 什么时候不适用

- 内部工程文档（CLAUDE.md、架构决策记录、设计文档）：可以提代码细节、内部假设、团队自嘲——读者是你和同事
- 技术博客 / Tutorial：有明确的"作者视角"需求
- Bug 追查日志：观察性措辞（"从日志看"、"抓包发现"）是合理的
- 给 AI 自己看的 memory：保留尽可能多的上下文，不用管对外稿规则
