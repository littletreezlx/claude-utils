# CLAUDE.md 体系设计决策

记录全局 CLAUDE.md 及指引体系的设计理念和关键决策，供未来维护参考。

## 核心设计理念

**"调度路由器"而非"文化手册"**：全局 CLAUDE.md 的定位是 Claude Code 的行为调度表，不是开发者教育手册。每条规则应包含明确的触发条件和动作，而非价值观陈述。

**信息密度 > 行数限制**：官方建议 < 200 行，但真正的指标是"指令密度"——含 If/Then 结构的有效指令占比。300 行全是高密度规则，比 100 行一半废话要好。

## 关键决策

### 1. 什么内容留在全局 CLAUDE.md

**保留标准**：删掉后 Claude 行为会明显改变的内容。

- **铁律**：逆直觉的设计决策（如"代码≠测试时信任测试"），Claude 不会自己这样做
- **自动触发规则表**：调度中枢，定义 Skill/Command 的自动触发条件
- **紧急停止机制**：安全护栏，防止 Claude 跑偏
- **AI 测试陷阱**：AI 特有的问题（自己出题自己答），非通用测试常识

**删除标准**：Claude 默认就会做的事。

- 认知框架（三段式思考）、调试分层诊断、并行工具调用 — 系统 prompt 已有
- getter 不需要测试、难测=难维护 — 通用编程常识
- 架构平衡启发式（选最简单方案）— 已被铁律 #4"务实主义"覆盖

### 2. Eager Load vs Lazy Load

| 加载方式 | 判断标准 | 示例 |
|---------|---------|------|
| Eager（`@path`） | Claude 高频引用，影响决策判断 | doc-structure.md（文档结构表） |
| Lazy（纯路径） | 低频参考，需要时才查阅 | testing.md（测试详细指引） |

**实证教训**：doc-structure.md 最初设计为 lazy load，结果导致 Claude 问"文档结构是否合理"时无法直接回答，反而触发了重型 Skill。改回 eager load 后恢复正常。

**结论**：不要盲目追求懒加载。判断标准是使用频率——Claude 经常需要用来做决策的内容，在上下文里比让它去找要好。

### 3. guides/ 目录定位

`~/.claude/guides/` 存放从 CLAUDE.md 拆分出的详细参考文档：

- **不是 rules/**：rules/ 是 Claude Code 原生的条件加载机制（支持 `paths:` frontmatter），适合文件类型特定规则
- **不是 docs/**：docs/ 是项目级文档，guides/ 是全局级指引
- **加载方式灵活**：通过 CLAUDE.md 中用 `@` 或纯路径引用来控制 eager/lazy

### 4. /claudemd audit 的分析深度

Audit 不仅仅是 checklist（检查行数、检查路径是否存在），还包含三个深层分析维度：

- **触发器明确度**：模糊指令（"尽量保持…"）→ 建议重构为条件指令（"当 X 时做 Y"）
- **指令冲突检测**：找出互相矛盾的规则对，要求用户设定优先级
- **ROI 评估**：`可执行性 × 违反损害度`，量化每条规则的保留价值
